## 1. 背景と目的

### 1.1 背景

**プロジェクトの背景:**

Astranoteは、**優れた執筆体験**と**Web小説に特化した高品質AI翻訳機能**を核とする多言語投稿プラットフォームです。[ユーザーリサーチ](../discovery/user_research.md)と[競合分析](../discovery/competitive_analysis.md)から、「高品質翻訳 × グローバル展開」の市場ポジションが完全に空白であることを発見しました。さらに、作家が抱える「3ツール横断の非効率な執筆環境（Notion → VS Code → カクヨム）」を統合する解決策も、既存プラットフォームには存在しません。

**解決したい課題:**

[リーンキャンバス](./lean_canvas.md)で定義した3つの課題のうち、最優先は以下です：

1. **執筆環境が分断されている** — Notion → VS Code → カクヨムの3ツール横断で「書く」以外の作業に時間を取られる（[HMW #4](../discovery/hmw.md)）
2. **翻訳品質が低すぎて読書体験が損なわれる** — 固有名詞の不一致、代名詞ミス、文脈喪失が頻発（[HMW #1](../discovery/hmw.md)）
3. **作家が自力で海外展開する手段がない** — 翻訳コスト・品質・時間の3つの壁（[HMW #3](../discovery/hmw.md)）

**参照すべきドキュメント:**
- [ペルソナ](../discovery/persona/index.md)
- [共感マップ](../discovery/empathy_map/index.md)
- [HMW](../discovery/hmw.md)
- [リーンキャンバス](./lean_canvas.md)

### 1.2 MVPの目的

**このMVPで検証したいこと:**
- [x] 統合執筆環境（プロット管理 + 設定参照 + エディタ + ワンクリック投稿）が、3ツール横断の非効率を解消し、作家の満足度を高めるか
- [x] AI翻訳（LLM + 固有名詞辞書）で、Web小説において「翻訳であることを忘れるレベル」の品質を達成できるか
- [x] 作家が固有名詞辞書を設定するだけで翻訳品質を大幅に改善できるか
- [x] 読者が翻訳品質に満足し、継続的に読書するか（リテンション）

**期待される学び:**
- 統合執筆環境が作家のワークフローにどの程度フィットするか（どの機能が最も使われるか）
- 「プロット・設定参照のサイドパネル」が実際にウィンドウ切り替えを減らすか
- AI翻訳の品質上限と課題（どのジャンル・要素で品質が落ちるか）
- 固有名詞辞書のカスタマイズがどの程度品質改善に寄与するか
- 読者が「許容できる翻訳品質」のライン（完璧を求めるか、意味が通じれば良いか）
- 1作品あたりの翻訳コスト（LLM API利用料）の実測値

## 2. 仮説定義

### 2.1 価値仮説（Value Hypothesis）

**価値仮説A（執筆体験）:**

**誰が:** 日本のWeb小説作家（[佐藤ユウキ](../discovery/persona/amateur_writer)のようなユーザー）

**何に困っていて:** Notion（プロット）→ VS Code（原稿）→ カクヨム（投稿）の3ツール横断で、「書く」以外の作業（ウィンドウ切り替え、コピペ、フォーマット修正）に時間を取られている

**何を提供すれば:** プロット管理・キャラクター設定参照・原稿執筆・ワンクリック投稿が1つの環境で完結する統合執筆環境を提供すれば

**どうなるか:** 作家は執筆に集中でき、ツール切り替えの非効率が解消され、プラットフォームに定着する（作家満足度4.0以上 / 5.0、執筆環境としてのNPS > 40）

**検証方法:** 5-10人の作家にMVPの統合エディタで1〜3章分を執筆してもらい、既存ワークフロー比での満足度とNPSを計測

**価値仮説B（翻訳品質）:**

**誰が:** 東アジアの小説（日本語のWeb小説）を英語で読みたいグローバル読者（[エミリー・チェン](../discovery/persona/global_reader)のようなユーザー）

**何に困っていて:** WebNovelのAI翻訳は「読めないレベル」で、固有名詞が章の途中で変わったり、文脈が失われたりして読書体験が損なわれている

**何を提供すれば:** Web小説に特化した高品質AI翻訳（固有名詞辞書 + ジャンル特化プロンプト + 文脈維持）を提供すれば

**どうなるか:** 読者は翻訳であることを忘れて物語に没頭でき、継続的に読書する（翻訳品質スコア4.0以上 / 5.0、7日後リテンション率 > 30%）

**検証方法:** 3〜5作品の日本語Web小説を翻訳し、50人以上の読者に読んでもらい、翻訳品質スコアとリテンション率を計測

### 2.2 成長仮説（Growth Hypothesis）

**どのチャネルで:** Reddit（r/noveltranslations、r/lightnovels）、Discord（翻訳小説ファンコミュニティ）

**どのように顧客を獲得し:** 人気作品の翻訳サンプルを投稿し、「この翻訳品質で読みたい方はこちら」とMVPサイトに誘導

**どのように収益化するか:** MVP段階では収益化しない。翻訳品質と読者の反応を検証することに集中

**検証方法:** Reddit/Discordでの投稿に対するエンゲージメント（upvote数、コメント数）、MVP登録への転換率を計測

### 2.3 最もリスクの高い仮説

**このMVPで最優先で検証すべき仮説（2つ）:**

> **仮説A: 統合執筆環境（プロット管理 + 設定参照サイドパネル + エディタ + ワンクリック投稿）が、3ツール横断の非効率を解消し、作家のプラットフォーム定着に繋がる**

**なぜこれがリスクが高いか:**
- 統合執筆環境は作家がAstranoteを「翻訳ツール」ではなく「メインの執筆環境」として選ぶ鍵。ここが弱いと、翻訳品質が高くても作家の定着率が低くなる
- 既存の強力なツール（VS Code、Notion）からの移行ハードルを超えるだけの価値があるか未検証
- 作家にとって「書く体験」は最も長時間触れる機能であり、UXの微妙な差が評価を大きく左右する

> **仮説B: AI翻訳（LLM + 固有名詞辞書 + ジャンル特化プロンプト）で、Web小説において翻訳品質スコア4.0以上（5.0満点）を達成できる**

**なぜこれがリスクが高いか:**
- Astranoteのグローバル展開が「翻訳品質の高さ」に依存している。この仮説が崩れると、海外展開のUVPが成立しない
- 既存のAI翻訳（WebNovelのMTL）は酷評されており、「AIで高品質翻訳」自体への市場の懐疑がある
- ファンタジー小説特有の固有名詞（魔法名、種族名、地名）が翻訳品質の最大の障壁

## 3. ターゲットユーザー

### 3.1 アーリーアダプター

**具体的なペルソナ:** [エミリー・チェン（グローバル読者）](../discovery/persona/global_reader)を代表とする、東アジア系コンテンツの翻訳小説ファン

**属性:**
- 年齢・性別: 20-35歳、男女問わず
- 職業: 学生〜社会人（デジタルネイティブ世代）
- 居住地: 英語圏（北米、欧州、オセアニア）
- デジタルリテラシー: 高い（Reddit/Discordを日常的に利用）

**行動特性:**
- r/noveltranslations や Discord の翻訳小説コミュニティに参加している
- WebNovelの翻訳品質に不満を持ちながらも代替がなく使っている
- NovelUpdatesで翻訳作品の更新を追跡している
- 新しいプラットフォームやツールを積極的に試す早期採用者

**獲得方法:** Reddit/Discordでの翻訳品質デモ投稿 → 興味を持ったユーザーをMVPサイトに誘導

**リクルーティング計画:**
- 目標人数: 読者50-100人、作家5-10人
- 期間: 2週間
- 方法: r/noveltranslationsでの告知投稿、Discord翻訳コミュニティでの口コミ、X.comでのWeb小説作家への直接DM

## 4. MVPタイプの選択

### 4.1 選択したMVPタイプ

**[ ] コンシェルジュ型** - 手動でサービスを提供
**[ ] ペーパーモック型** - プロトタイプで反応確認
**[ ] ランディングページ型** - LPで需要確認
**[ ] オズの魔法使い型** - 自動化に見せて手動対応
**[x] シングル機能型** - 1つの機能に絞った実装
**[ ] その他:**

### 4.2 選択理由

シングル機能型を選択した理由：

1. **検証の核心が「翻訳品質」というプロダクトの実物に依存** — ペーパーモックやランディングページでは翻訳品質を体験できない。読者が実際に翻訳された小説を読み、品質を評価する必要がある
2. **技術的実現可能性の検証が必要** — AI翻訳の品質が仮説通りに達成できるか、実際にLLMを動かして確認する必要がある
3. **コスト構造の検証** — 1作品あたりの翻訳コスト（LLM API利用料）を実測し、ビジネスモデルの財務的成立性を確認する
4. **コンシェルジュ型では不要な人力翻訳コストが発生** — AI翻訳の品質自体が検証対象なので、手動翻訳は本質的な検証にならない

### 4.3 期間と予算

**開発期間:** 6週間

**予算:** 70万円以内
- LLM API利用料: 〜15万円（翻訳処理）
- インフラ費用: 〜5万円（クラウドサーバー、ドメイン）
- エディタライブラリ: 〜5万円（Tiptap Pro等のリッチテキストエディタ）
- その他: 〜5万円（テストユーザーへのインセンティブ等）
- 人件費: 自社リソースで対応

**リソース:**
- 開発: 1-2名（フルスタック）
- デザイン: 0.5名（エディタUI + 読書ビューア）
- その他: コンテンツ（作家との翻訳許諾調整）

### 5.2 MVPに含めない機能（Will Not Have）

**意図的に含めない機能:**
- サブスクリプション / 課金機能（MVP段階では全機能無料）
- コミュニティ機能（コメント、フォロー、通知等）
- 多言語対応（英語以外の翻訳）
- モバイルアプリ（Webのみ）
- 検索 / レコメンデーション
- 投げ銭 / 作家への収益分配
- コンテンツ審査（AI生成コンテンツ検出）

**理由:**
これらは翻訳品質と執筆環境の検証には不要です。検証すべき仮説（翻訳品質・統合執筆環境）に絞り、その他のユーザー体験の付加価値は次のイテレーション以降で追加します。

### 5.3 次のイテレーションで検討する機能

- サブスクリプション（Free / Premium）
- 作家ダッシュボード（言語圏別読者分析）
- 読者コメントの自動翻訳配信
- 多言語展開（中国語、韓国語、スペイン語）
- 投げ銭機能
- モバイル対応（PWA / ネイティブアプリ）
- エディタの高度な機能（テンプレート、アウトライン、バージョン管理）
- 設定資料と原稿の自動リンク（キャラ名をクリックすると設定表示）

  - 指標: 読者が自発的にSNS/Discordでシェアした回数
  - 目標: 10件以上（オーガニック）

### 6.2 定性指標

**ユーザーインタビュー目標:**
- 実施人数: 10-15人（読者8-10人、作家3-5人）
- 期間: MVP公開後2-3週間
- 確認したいこと:
  - 翻訳品質のどの要素（固有名詞、文体、文脈維持）が最も重要か
  - WebNovelの翻訳と比べてどう感じたか（具体的な比較）
  - 「翻訳であることを忘れた瞬間」はあったか
  - 作家が固有名詞辞書を設定するUXは直感的だったか
  - 有料化した場合に課金意欲があるか（WTP: Willingness to Pay）

**観察したいユーザー行動:**
- 翻訳品質スコアが低い章で読者が離脱するか（品質 → リテンションの因果関係）
- 固有名詞辞書を設定した作品と未設定の作品で品質スコアに差があるか
- 読者がフィードバックを積極的に残すか（品質改善ループの成立可能性）

### 6.3 成功/失敗の判断基準

**成功と判断する条件（すべて満たす場合）:**
- 作家の執筆環境満足度が4.0以上（作家5人以上の平均）
- 翻訳品質スコアが4.0以上（読者50人以上の平均）
- 7日後リテンション率が30%以上
- 作家5人以上が「この環境で書き続けたい」かつ「この品質なら作品を公開したい」と回答
- 1章あたりの翻訳コストが$0.50以下（財務的に成立）

**失敗と判断する条件（いずれか1つでも該当する場合）:**
- 作家の執筆環境満足度が3.0以下（既存ツールの方が良いと判断される）
- 翻訳品質スコアが3.5以下（WebNovelと大差ないレベル）
- 7日後リテンション率が15%以下（翻訳品質では読者を引き留められない）
- 1章あたりの翻訳コストが$2.00以上（財務的に不成立）

**判断のタイミング:** MVP公開後4週間後

## 7. 検証計画（MVPキャンバス）

### 7.1 仮説

**最も重要な仮説:**

> A: 統合執筆環境（プロット管理 + 設定参照サイドパネル + エディタ + ワンクリック投稿）が、3ツール横断の非効率を解消し、作家のプラットフォーム定着に繋がる
>
> B: AI翻訳（LLM + 固有名詞辞書 + ジャンル特化プロンプト + 章間文脈維持）で、Web小説において翻訳品質スコア4.0以上を達成でき、読者は継続的に読書する

### 7.2 リスク

**想定されるリスク:**
1. **統合エディタがVS Code/Notionの代替として不十分** — 作家は既に高機能なツールに慣れており、Webエディタでは機能不足と感じる可能性
2. **翻訳品質が目標に達しない** — LLMの翻訳品質がファンタジー小説の特殊な用語に対応できない可能性
3. **翻訳コストが高すぎる** — 長編小説（100章以上）の翻訳コストがサブスクリプション収益を上回る可能性
4. **作家の許諾が得られない** — MVP用のコンテンツ（翻訳対象の日本語小説）を確保できない可能性
5. **テストユーザーが集まらない** — Reddit/Discordからの誘導が十分に機能しない可能性

**対応策:**
- リスク1: 「VS Codeの代替」ではなく「プロット・設定・投稿の統合」にフォーカス。MVPでは必要最小限のエディタ機能に絞り、差別化ポイント（サイドパネル参照、ワンクリック投稿）を優先実装
- リスク2: 複数のLLMモデル（Claude, GPT-4等）を比較検証。翻訳プロンプトをイテレーティブに改善
- リスク3: 翻訳の前処理（チャンク分割、キャッシュ活用）でコストを最適化。最悪ケースのコストも事前に算出
- リスク4: カクヨムのCC BY-NC作品を使用、または自ら短編を執筆。協力作家への早期アプローチ
- リスク5: 翻訳品質のサンプルを先にRedditに投稿し、反応を見てからMVPの告知を行う

### 7.3 検証方法

**実施内容:**
- [x] 日本語Web小説3-5作品（各10-20章）をAI翻訳エンジンで翻訳
- [ ] 固有名詞辞書あり/なしの2パターンで翻訳品質を比較
- [ ] MVPサイトで翻訳済み作品を公開し、50-100人の読者に読んでもらう
- [ ] 翻訳品質スコア、リテンション率、読了率を4週間計測
- [ ] 読者10-15人、作家3-5人にインタビューを実施
- [ ] 1章あたりの翻訳コストを算出し、財務シミュレーションを実施

**期間:** 開発6週間 + 検証4週間 = 合計10週間

**必要なリソース:** エンジニア1-2名、LLM APIアクセス、クラウドサーバー、リッチテキストエディタライブラリ、テスト用コンテンツ

### 7.4 計測するデータ

**定量データ:**
- 翻訳品質スコア（章ごと、5段階、読者評価）
- ページビュー数（章ごと、日次）
- リテンション率（1日後、3日後、7日後、14日後、30日後）
- 読了率（第1章→最新章の到達率）
- LLM API利用料（章ごと、モデルごと）
- Reddit/Discord投稿のエンゲージメント（upvote、コメント、CTR）

**定性データ:**
- ユーザーインタビューの録音・要約
- 翻訳品質フィードバックの自由記述コメント
- Reddit/Discordでの自発的な感想・共有
- 作家の固有名詞辞書設定時の行動観察

**計測ツール:**
- アナリティクス: Plausible Analytics（プライバシー重視、軽量）
- ヒートマップ: 不要（MVPでは読書ビューアのみ）
- フィードバック収集: 自作の評価ウィジェット（5段階 + 自由記述）
- その他: LLM API利用量ダッシュボード（自作）

## 8. 開発計画

### 8.1 技術スタック

**フロントエンド:** Next.js（React）+ Tailwind CSS
- 理由: SSRで高速な初回表示、読書ビューアのカスタマイズが容易

**バックエンド:** Next.js API Routes + Prisma ORM
- 理由: フロントエンドと同一リポジトリで管理でき、開発速度を最大化

**インフラ:** Vercel（フロントエンド） + Supabase（PostgreSQL + Auth）
- 理由: 無料ティアで十分、セットアップが最小限

**AI翻訳:** Anthropic Claude API（メイン） + OpenAI GPT-4（比較用）
- 理由: Claudeの長文コンテキスト処理能力がWeb小説翻訳に適している

**ツール:** GitHub（コード管理）、Plausible（アナリティクス）

**選定理由:** MVPの開発速度を最大化し、4週間以内にリリースできるスタックを選択。本番スケール時に技術的負債が少ない構成。

### 8.2 システム構成

```
[作家] → 統合エディタ（プロット・設定・原稿） → [Next.js API]
              ↕                                       ↓
    [設定管理サイドパネル]                    [固有名詞辞書DB]
    （キャラ設定・世界観）                          ↓
                                            [LLM API (Claude)]
                                                    ↓
                                            [翻訳済みテキスト保存]
                                                    ↓
[読者] ← 読書ビューア ← [Next.js SSR] ← [Supabase DB]
                    ↓
            [品質フィードバック] → [Supabase DB]
```

### 8.4 技術的な制約・前提

**制約:**
- LLM APIの入力トークン制限（1回のリクエストで1章分を翻訳できない場合、チャンク分割が必要）
- LLM APIのレート制限（大量の翻訳を短時間で実行できない可能性）
- 翻訳の一貫性（章をまたいだ固有名詞・文体の統一はプロンプト設計で対応）

**前提:**
- LLM APIの翻訳品質は、適切なプロンプト設計と固有名詞辞書の提供で十分な品質に達する（仮説）
- カクヨムのCC BY-NC作品、または協力作家の作品をMVP用コンテンツとして使用できる
- Reddit/Discordコミュニティからの流入で50人以上のテストユーザーを獲得できる

### 8.5 品質基準

**最低限の品質基準:**
- [x] 統合エディタでリッチテキスト（ルビ、傍点、太字、見出し）が正常に動作する
- [x] サイドパネルからキャラクター設定・世界観設定を即座に参照できる
- [x] エディタからワンクリックで投稿が完了する（コピペ不要）
- [x] 翻訳された小説が意味の通る英語になっている（MTLレベルではない）
- [x] 固有名詞が全章を通じて一貫している
- [x] 読書ビューアが主要ブラウザ（Chrome、Safari、Firefox）で動作する
- [x] ページ読み込みが3秒以内
- [x] モバイルブラウザでも読書可能（レスポンシブ対応）

**テスト方針:** 統合エディタは作家3-5人による実地テスト（実際に1-3章分を執筆してもらう）。翻訳品質は手動QA（翻訳結果を人間がレビュー）。アプリケーションは基本的な動作確認とクロスブラウザテスト。
