# 📖 使い方ガイド

## ユーザーストーリーとは

ユーザーストーリーは、**ユーザーの視点から記述された機能要求**です。技術的な仕様ではなく、「誰が、何を、なぜ必要とするか」を簡潔に表現することで、チーム全体が共通の理解を持てるようにします。

### ユーザーストーリーの3つの要素

```
ユーザーとして、        → 誰が（Who）
[何かをしたい]          → 何を（What）
なぜなら、[理由]だから  → なぜ（Why）
```

**英語形式:**
```
As a [role]
I want to [feature]
So that [benefit]
```

## なぜユーザーストーリーを書くのか

**1. ユーザー中心の開発**
- 技術ではなく、ユーザーの価値に焦点を当てる
- 「作りたいもの」ではなく「必要なもの」を明確化

**2. コミュニケーションツール**
- 開発者、デザイナー、PM、ビジネス側が共通言語で話せる
- 要件の背景と目的を理解しやすい

**3. 優先順位付けの基盤**
- ビジネス価値が明確なので、優先順位をつけやすい
- ROIの高いストーリーから着手できる

**4. テスト可能な単位**
- 受け入れ基準を定義することで、完了の定義が明確
- テストケースに変換しやすい

## ユーザーストーリーの書き方

### ステップ1: ペルソナを明確にする
[Discovery のペルソナ](../discovery/persona.md)を参照し、誰のためのストーリーかを特定します。

### ステップ2: ペインポイントを特定
[ジャーニーマップ](./journey_map.md)から、解決すべき課題を抽出します。

### ステップ3: ユーザーストーリーを記述
以下の形式で記述します:

```
ユーザーとして、
[何かをしたい]
なぜなら、[理由]だから
```

### ステップ4: 受け入れ基準を定義
ストーリーが完了したと判断できる基準をチェックリスト形式で記述します。

- [ ] 基準1
- [ ] 基準2
- [ ] 基準3

### ステップ5: 優先順位とポイントを設定
- **優先度**: ビジネス価値と緊急度から判断
- **ストーリーポイント**: フィボナッチ数列（1, 2, 3, 5, 8, 13）で見積もり

### ステップ6: テストケースに変換
[テストケース（Gherkin）](./test_cases.md)で、受け入れ基準を実行可能なテストに変換します。

## エピックとストーリーの関係

### エピック
大きな機能やテーマをまとめたもの。複数のユーザーストーリーで構成されます。

**エピックの例:**
- ユーザー登録機能
- 決済機能
- ダッシュボード機能

### ストーリー
エピックを実装可能な小さな単位に分割したもの。

## 受け入れ基準の書き方

受け入れ基準は、**ストーリーが完了したと判断するための具体的な条件**です。

### Given-When-Then 形式

```
Given [前提条件]
When [アクション]
Then [期待結果]
```

**チェックリスト形式:**
- [ ] ユーザーがメールアドレスとパスワードを入力できる
- [ ] 「登録」ボタンをクリックするとアカウントが作成される
- [ ] ウェルカムメールが送信される
- [ ] ダッシュボードにリダイレクトされる

## ストーリーマッピングへ

ユーザーストーリーをリリース計画に落とし込むには、[ストーリーマッピング](./story_mapping.md)を活用します。ストーリーマッピングでは、ユーザーの流れ（横軸）と優先順位（縦軸）の2次元でストーリーを可視化し、MVP から段階的なリリース計画を立てることができます。

## ジャーニーマップから次のステップへ

[ジャーニーマップ](./journey_map.md)で特定したペインポイントと改善機会を、実装可能なユーザーストーリーに変換し、プロダクト改善につなげます。

### ステップ1: ペインポイントを特定

ジャーニーマップの「ペインポイント」行から、ユーザーが困っている箇所を抽出します。

### ステップ2: 改善機会をストーリーに変換

ペインポイントと改善機会を、ユーザーストーリー形式で記述します。

**ユーザーストーリーのフォーマット:**
```
ユーザーとして、
[何かをしたい]
なぜなら、[理由]だから
```

**変換例:**

| ペインポイント | 改善機会 | ユーザーストーリー |
|--------------|---------|------------------|
| 登録フォームが長くて離脱する | ソーシャルログイン追加 | ユーザーとして、Googleアカウントで登録したい、なぜなら長いフォーム入力を避けたいから |
| 使い方が分かりにくい | インタラクティブなチュートリアル | ユーザーとして、初回利用時にガイドを見たい、なぜなら早く使い方を理解したいから |
| 一部の機能が遅い | パフォーマンス最適化 | ユーザーとして、レポートを3秒以内に表示したい、なぜなら効率的に作業したいから |

### ステップ3: 受け入れ基準を定義

各ユーザーストーリーに対して、完了の定義を明確にします。

### ステップ4: 優先順位とポイントを設定

- **優先度**: ペインポイントのインパクト × 改善の実現可能性
- **ストーリーポイント**: 相対見積もり（フィボナッチ数列: 1, 2, 3, 5, 8, 13）

| ストーリー | 優先度 | ビジネス価値 | ポイント |
|-----------|-------|-------------|---------|
| Googleログイン | 高 | 8 | 5 |
| チュートリアル | 高 | 9 | 8 |
| レポート最適化 | 中 | 7 | 5 |

### ステップ5: [テストケース（Gherkin）](./test_cases.md)に変換

受け入れ基準を、実行可能なテストケースに変換します。

### ステップ6: 改善施策の実行

- 優先度の高いストーリーから着手
- スプリント/イテレーションで計画的に実装
- A/Bテストで効果を検証
- KPIで成果を測定

### ステップ7: 継続的な更新

- プロダクトリリース後、ジャーニーマップを更新
- 新しいペインポイントや改善機会を追加
- ストーリーバックログに反映
- チームでレビューを定期実施

## 次のステップ

ユーザーストーリーを作成したら、以下のステップでリリース計画とテスト化を進めます。

1. **[ストーリーマッピング](./story_mapping.md)を作成**
   - ストーリーを2次元マップで可視化
   - エピックとリリース計画を策定
   - MVPを明確化

2. **スプリント計画**
   - ストーリーポイントとベロシティを確認
   - スプリントにストーリーを割り当て
   - チームで見積もりと計画を合意

3. **[テストケース（Gherkin）](./test_cases.md)を作成**
   - 受け入れ基準をGherkin形式で記述
   - 自動テストの基盤を準備

4. **開発と実装**
   - テスト駆動開発（BDD）で実装
   - 継続的インテグレーション

## まとめ

ユーザーストーリーは、**ユーザー視点での価値を中心に据えた開発**を実現するための強力なツールです。

**このドキュメントの使い方:**
1. 上部のテンプレートに実際のユーザーストーリーを記入
2. ジャーニーマップからペインポイントを抽出
3. INVEST原則に従ってストーリーを記述
4. 受け入れ基準を明確化
5. 優先順位付けとリリース計画
6. テストケースに変換して品質保証
