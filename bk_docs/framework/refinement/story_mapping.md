# 📖 使い方ガイド

## ストーリーマッピングとは

ストーリーマッピングは、**ユーザーストーリーを2次元の地図として可視化する手法**です。ジェフ・パットン（Jeff Patton）が提唱したこの手法により、ユーザーの流れ（横軸）と優先順位（縦軸）を同時に把握でき、チーム全体で共通のプロダクトビジョンを持つことができます。

### ストーリーマップの2つの軸

**横軸（左から右）:**
- ユーザーの時系列での活動の流れ
- ユーザージャーニーに沿った体験の順序

**縦軸（上から下）:**
- 優先順位（上にあるほど重要）
- リリース計画（水平線でグループ化）

## なぜストーリーマッピングが必要か

**1. プロダクト全体を俯瞰できる**
- バックログの羅列ではなく、ユーザー体験の流れで理解できる
- 木を見て森も見る

**2. リリース計画が立てやすい**
- MVPに必要な最小限のストーリーを特定
- 段階的なリリース戦略を可視化

**3. チームの共通理解**
- PM、デザイナー、エンジニア全員が同じ地図を見る
- プロダクトのゴールを共有

**4. 優先順位付けが明確**
- 「何を今やるべきか」が一目瞭然
- ユーザー価値に基づいた判断

**5. ギャップの発見**
- ユーザー体験の抜け漏れを見つけやすい
- 全体の整合性を保つ

## ストーリーマップの構造

### 基本構造

```
┌─────────────────────────────────────────────────────────┐
│ エピック（大きな活動）                                    │  ← ユーザーの活動
├─────────────────────────────────────────────────────────┤
│ ストーリー（詳細なタスク）                                │  ← 実装する機能
│ ストーリー                                               │  ← 優先度順
│ ストーリー                                               │
└─────────────────────────────────────────────────────────┘
  横方向 = ユーザーの流れ
  縦方向 = 優先度（上が高い）
```

### 要素の説明

**エピック（最上部の行）:**
- ユーザーの大きな活動やゴール
- 例: 「アカウント登録」「商品検索」「購入」「配送追跡」

**ストーリー（エピックの下に縦に並ぶ）:**
- エピックを実現するための具体的な機能
- 上から優先度が高い順に配置
- 例:   - エピック「アカウント登録」の下に:
    - メールアドレスで登録（高優先度）
    - Googleログイン（中優先度）
    - プロフィール画像追加（低優先度）

**リリースライン（水平線）:**
- ストーリーを横断する線でリリースをグループ化
- 最上部の線下 = MVP
- 2本目の線下 = リリース2
- 以降同様

## 作成手順

### ステップ1: [ジャーニーマップ](./journey_map.md)を参照

ユーザーの活動の流れを理解します。

**ジャーニーマップのフェーズ → エピック:**
- 認知 → 「サービスを知る」
- 検討 → 「機能を確認する」
- 登録 → 「アカウント作成」
- 初回利用 → 「初期設定」
- 継続利用 → 「日常的に使う」

### ステップ2: エピックを横に並べる

ユーザーの時系列での活動順に、エピックを左から右へ配置します。

**Tips:**
- 付箋やホワイトボードを使うと効果的
- デジタルツール: Miro, Mural, Jira などを活用

### ステップ3: エピックをストーリーに分解

各エピックの下に、実装可能な小さなユーザーストーリーを書き出します。

**例（エピック: アカウント作成）:**
- メールアドレスとパスワードで登録
- メール認証
- Googleアカウントで登録
- GitHubアカウントで登録
- プロフィール情報入力
- プロフィール画像アップロード

### ステップ4: 優先順位で並べ替える

各エピック内で、ストーリーを優先度順に縦に配置します。

**優先順位の判断基準:**
- ユーザー価値の高さ
- ビジネスインパクト
- 技術的依存関係
- リスクの大きさ

### ステップ5: リリースラインを引く

水平線を引いて、リリースごとにストーリーをグループ化します。

**MVP（最初のリリース）:**
- 最上部の行（各エピックで最も重要なストーリー）
- ユーザーがエンドツーエンドで価値を得られる最小セット

**リリース2以降:**
- 段階的に機能を追加
- ユーザーフィードバックを元に調整

### ステップ6: [ユーザーストーリー](./user_stories.md)を詳細化

マップ上の各ストーリーについて、詳細を記述します。

- ユーザーストーリーの形式で記述
- 受け入れ基準を定義
- ストーリーポイントを見積もり

### ステップ7: [テストケース](./test_cases.md)を作成

各ストーリーのテストケースを Gherkin 形式で記述します。

## 次のステップ

ストーリーマッピングを作成したら、以下のステップで実装とテストを進めます。

1. **スプリント計画**
   - ベロシティに基づいてストーリーをスプリントに割り当て
   - チームのキャパシティを考慮
   - リリース日程を調整

2. **[テストケース](./test_cases.md)を作成**
   - 優先度の高いストーリーからGherkin形式で記述
   - 受け入れ基準を実行可能なテストに変換
   - 自動テストの準備

3. **開発と実装**
   - テスト駆動開発（BDD）で実装
   - 継続的インテグレーション

4. **定期的な見直し**
   - スプリントレビュー後にマップを更新
   - ユーザーフィードバックを反映
   - 優先順位を再評価
