## プロダクトディスカバリーとは

プロダクトディスカバリーは、**何を作るかを決定するための反復的なプロセス**です。顧客の真のニーズや課題を深く理解し、それらを満たす価値ある製品やソリューションを発見することを目的としています。

「それ、本当？」という問いをチーム全員で繰り返し問いかけ、思い込みや想像ではなく、**事実に基づく意思決定**を実現します。

### ディスカバリー vs デリバリー

- **ディスカバリー（Discovery）**: 何を作るかを決定する活動
- **デリバリー（Delivery）**: 生産品質のプロダクトを作り、ユーザーへ届け、メンテナンスする活動

ディスカバリーが不十分だと、「開発した機能が使われない・伸びない」という問題が発生します。

## なぜディスカバリーが重要か

徹底的なプロダクトディスカバリーを行うことで、以下が実現できます。

**1. 無駄なコストの削減**
- 使われない機能の開発を防ぐ
- 時間・人員・リソースを最適化

**2. 賢明な意思決定**
- ユーザーが実際に何を望んでいるか客観的に理解
- 時間とリソースをどこに集中させるか判断できる

**3. 早期の課題発見**
- プロダクト開発プロセスの早い段階で潜在的な課題を特定
- リスクを低減し、成功確率（打率）を向上

**4. チームの疲弊防止**
- 根拠のある判断に基づく開発により、無駄な実装を減らす
- チームの自律的な意思決定を支える

## ディスカバリーの主要な活動

優れたプロダクトディスカバリーチームは、以下の2つの重要な活動を**毎週**行っています。

**1. 顧客へのインタビュー**
- 機会を発見するのに役立つ
- ユーザーの真のニーズや課題を理解

**2. 仮説検証**
- 解決策を発見するのに役立つ
- アイデアの妥当性を早期に確認

## このセクションで扱うツール

プロダクトディスカバリーには数多くの戦術やフレームワークがありますが、このセクションでは特に重要な5つのツールを扱います。

### 1. [ユーザーリサーチ（User Research）](./user_research.md)

**目的:** ユーザーの生の声を収集し、事実に基づくインサイトを得る

**使うタイミング:**
- プロジェクト開始時（ペルソナ作成の前）
- 思い込みや仮説を検証したい時
- ユーザーの本音を知りたい時

**主な手法:**
- アプリストアレビュー分析（App Store、Google Play）
- SNS分析（X.com、Reddit、note等）
- ユーザーインタビュー
- アンケート調査
- コミュニティ・フォーラム観察

**期待される効果:**
- 思い込みではなく事実に基づく意思決定
- ユーザーの本当のペインポイントの発見
- ペルソナや共感マップの根拠となるデータの収集

### 2. [競合分析（Competitive Analysis）](./competitive_analysis.md)

**目的:** 競合製品の強み・弱みを理解し、差別化ポイントを発見

**使うタイミング:**
- プロジェクト開始時（市場理解のため）
- ユーザーリサーチと並行して実施
- 機能優先度を決定する時

**主な手法:**
- 機能比較表の作成
- 競合アプリのレビュー分析
- SNS活動の観察
- 価格・ビジネスモデル比較
- SWOT分析
- ポジショニングマップ

**期待される効果:**
- 市場の全体像の把握
- 競合の弱点（自社のチャンス）の発見
- 独自の価値提案（UVP）の明確化
- やるべきこと・やらないことの判断

### 3. [ペルソナ（Persona）](./persona.md)

**目的:** ターゲットユーザーを代表する架空の人物像を作成

**使うタイミング:**
- プロジェクト初期のユーザー定義
- チーム全体で共通のユーザーイメージを持つ必要がある時

**期待される効果:**
- ユーザー視点での意思決定
- チーム全体での認識統一

### 4. [共感マップ（Empathy Map）](./empathy_map/index.md)

**目的:** ユーザーの感情や思考を視覚的にマッピング

**使うタイミング:**
- ペルソナ作成後、より深くユーザーを理解したい時
- ユーザーインタビューやリサーチ結果の整理

**期待される効果:**
- ユーザーの潜在ニーズの発見
- 表面的な要望の奥にある真のニーズの理解

**構成要素:**
- 考えていること・感じていること
- 見ていること
- 言っていること・していること
- 聞いていること
- 悩み・ストレスになっていること（Pain）
- あったら嬉しいこと（Gain）

### 5. [HMW（How Might We）](./hmw.md)

**目的:** 課題を「どうすれば〜できるだろうか？」という形式で再定義し、アイデア発想を促進

**使うタイミング:**
- 課題やペインポイントが明確になった後
- アイデア創出の前（定義フェーズの終わり）

**期待される効果:**
- ネガティブな課題をポジティブな機会に転換
- 多様な解決策を引き出す
- チーム全体で同じ方向性を共有しながら創造的に発想

## 基本的なディスカバリープロセス

```
1. テーマ設定
   ↓
   事業計画やプロダクトビジョンをベースに、
   ディスカバリーするテーマを決定

2. ユーザー理解
   ↓
   ・ペルソナの作成
   ・ユーザーインタビュー
   ・共感マップの作成

3. 課題の特定
   ↓
   ・ペインポイントの整理
   ・インサイトの発見

4. 機会の探索
   ↓
   ・HMWによる課題の再定義
   ・複数角度からの問いかけ

5. アイデア創出
   ↓
   ・ブレインストーミング
   ・解決策の発散

6. 仮説検証
   ↓
   ・プロトタイピング
   ・ユーザーテスト
   ・A/Bテスト

7. 意思決定
   ↓
   データに基づき、何を作るかを決定

（2〜7を反復）
```

## ディスカバリーで大切なこと

**「それ、本当？」と問い続ける**
- 思い込みや想像に基づく判断を排除
- 常に根拠を求める姿勢

**顧客の事実のハコを用意する**
- ユーザーインサイトのデータベース（SSoT）
- 各職種の自律的な意思決定を支える

**既知と未知を明確にする**
- チームで何が分かっていて、何が分かっていないかを合意
- 特に「既知の未知（疑問）」の解像度を高める

**毎週実践する**
- 顧客インタビューと仮説検証を継続的に実施
- 一度きりではなく、反復的なプロセスとして定着

## その他のディスカバリー手法

このセクションでは扱っていませんが、以下のような手法も有効です。

- カスタマージャーニーマッピング
- ユーザーストーリーマッピング
- ジョブ理論
- 機会解決ツリー
- インパクトマッピング
- ユーザビリティテスト
- デマンドテスト
- エスノグラフィ調査

## はじめ方

1. **[ユーザーリサーチ](./user_research.md)と[競合分析](./competitive_analysis.md)で事実を集める**
   - アプリレビュー、SNS、インタビューでユーザーの声を収集
   - 競合の強み・弱みを把握
   - ペインポイントとニーズを明確化
   - 思い込みではなく事実に基づく

2. **[ペルソナ](./persona/index.md)を作成する**
   - リサーチ結果を基にターゲットユーザーを定義
   - 実データに基づいて作成

3. **[共感マップ](./empathy_map/index.md)で深掘りする**
   - ペルソナの感情や思考を可視化
   - ペインとゲインを特定

4. **[HMW](./hmw.md)で機会を探る**
   - 課題を問いかけに変換
   - アイデア創出の準備

5. **アイデアを出して、検証する**
   - 仮説を立てる
   - 小さく早く検証
   - 次は[Validation（検証）](../validation/)セクションへ

6. **反復する**
   - 学びを次のサイクルに活かす
