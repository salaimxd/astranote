# AIプロンプトテンプレート集

> [!NOTE]
> 生成AIを使ってプロダクトドキュメント作成・レビュー・管理を効率化するためのプロンプトテンプレート集です。
> Claude、ChatGPT、GitHub Copilotなど、任意のAIツールで使用できます。

---

## 📝 ドキュメント作成支援

### PRD（要件定義書）作成

```
以下の情報を基にPRDを作成してください：

**機能名**: [機能名を記入]
**背景**: [なぜこの機能が必要か]
**ターゲットユーザー**: [Personaへのリンクまたは説明]
**ビジネス目標**: [達成したいKPI]

テンプレートに従って作成してください：
- 参照: [プロジェクトパス]/docs/30_Definition_Requirements/31.01_PRD_<Function_Name>.md

以下の点を必ず含めてください：
- Must/Should/Could/Won't Haveの明確な分類
- 測定可能な成功指標（KPI）
- ユーザーストーリー（As a [role], I want to [action], so that [benefit]）
- エッジケース（エラー、空状態、権限不足）
- セキュリティ/プライバシー考慮事項
```

### Tech Spec（技術仕様書）作成

```
以下のPRDを基にTech Specを作成してください：

**PRD**: [PRDへのリンクまたは内容を貼り付け]

テンプレートに従って作成してください：
- 参照: [プロジェクトパス]/docs/50_Engineering_Tech/51.01_TechSpec_<Function_Name>.md

以下の技術的側面を詳細に記述してください：
- アーキテクチャ概要（図を含む）
- データモデル（テーブル設計）
- APIエンドポイント（リクエスト/レスポンス）
- セキュリティ実装（認証、認可、暗号化）
- パフォーマンス要件（レスポンスタイム、スループット）
- テスト戦略（Unit/Integration/E2E）
- モニタリング/ログ戦略
```

### Persona（ペルソナ）作成

```
以下のユーザーインタビュー結果からペルソナを作成してください：

**インタビュー結果**: [User Interviews Summaryの内容を貼り付け]

テンプレートに従って作成してください：
- 参照: [プロジェクトパス]/docs/20_Discovery_Research/21.02_Persona_<Persona_Name>.md

以下の要素を含めてください：
- 基本情報（名前、年齢、職業、企業規模）
- 背景とゴール
- ペインポイント（具体的な困りごと）
- 利用シーン（1日の典型的なワークフロー）
- 意思決定プロセス
- 競合製品の利用状況
```

### Release Notes（リリースノート）作成

```
以下のPRDとTech Specからユーザー向けRelease Notesを作成してください：

**PRD**: [リンク]
**Tech Spec**: [リンク]
**リリース日**: [YYYY/MM/DD]

テンプレートに従って作成してください：
- 参照: [プロジェクトパス]/docs/30_Definition_Requirements/32.01_Release_Notes.md

以下の形式で記述してください：
- 新機能: ユーザー価値を強調（技術詳細は避ける）
- 改善点: Before/Afterを明確に
- バグ修正: 影響範囲を明記
- 既知の問題: 回避策を提示

トーン: 明るく、分かりやすく、専門用語を避ける
```

## 🔍 ドキュメントレビュー支援

### 整合性チェック

```
以下の2つのドキュメントの整合性をチェックしてください：

**PRD**: [PRDの内容を貼り付け]
**Tech Spec**: [Tech Specの内容を貼り付け]

以下の観点でチェックしてください：
1. **要件の漏れ**: PRDで定義された機能がTech Specで実装されているか
2. **矛盾**: PRDとTech Specで異なる仕様が記載されていないか
3. **不足**: セキュリティ、パフォーマンス、エラーハンドリングが考慮されているか
4. **技術的実現可能性**: PRDの要件が技術的に実装可能か
5. **スコープクリープ**: Tech SpecがPRDのスコープを超えていないか

結果を以下の形式で出力してください：
- ✅ 整合性OK
- ⚠️ 要確認（理由）
- ❌ 矛盾あり（詳細）
```

### PRD品質レビュー

```
以下のPRDを品質レビューしてください：

[PRDの内容を貼り付け]

評価基準（各2点、合計10点）：
1. **明確性**: 目的とスコープが明確か
2. **測定可能性**: KPIが定量的か
3. **完全性**: Must/Should/Could/エッジケースが網羅されているか
4. **整合性**: 他ドキュメント（Persona、Roadmap）との矛盾がないか
5. **実装可能性**: エンジニアが実装を開始できるレベルか

各項目を採点し、具体的な改善提案を出してください。

出力形式：
## 評価結果
- 明確性: X/2点 - [理由]
- ...

## 改善提案
1. [具体的な提案]
2. ...
```

### Tech Spec品質レビュー

```
以下のTech Specを品質レビューしてください：

[Tech Specの内容を貼り付け]

チェック項目：
- [ ] アーキテクチャ図が明確か
- [ ] データモデルがER図で表現されているか
- [ ] APIエンドポイントが全て定義されているか
- [ ] セキュリティ脆弱性が考慮されているか（OWASP Top 10）
- [ ] パフォーマンス要件が数値で定義されているか
- [ ] エラーハンドリングが網羅されているか
- [ ] テスト戦略が明確か（カバレッジ目標）
- [ ] モニタリング/アラートが定義されているか
- [ ] ロールバック手順が記載されているか

不足している項目と具体的な追加提案を出してください。
```

## 📊 ドキュメント要約・分析

### 3行要約（エンジニア向け）

```
以下のPRDを3行で要約してください：

[PRDの内容を貼り付け]

対象読者: エンジニア
フォーカス: 技術的な実装要件とスコープ

出力形式：
1. [何を実装するか]
2. [主要な技術的制約]
3. [重要なマイルストーン]
```

### エグゼクティブサマリー作成

```
以下のドキュメントから経営層向けのエグゼクティブサマリーを作成してください：

**PRD**: [リンク]
**Market Trends**: [リンク]
**Launch Plan**: [リンク]

以下の要素を含めてください（各2-3文）：
- ビジネス機会（なぜ今この機能が必要か）
- 期待される成果（KPI、売上インパクト）
- リスクと緩和策
- 投資規模（開発期間、リソース）
- 競合との差別化

長さ: 1ページ以内（500文字程度）
```

### 変更影響分析

```
以下のドキュメントを変更します：

**変更対象**: [ドキュメント名]
**変更内容**: [変更の概要]

プロジェクト構造を確認した上で、影響を受ける可能性のあるドキュメントをリストアップしてください。

出力形式：
## 影響を受けるドキュメント
1. **[ドキュメント名]** (優先度: High/Medium/Low)
   - 影響内容: [説明]
   - 確認・更新すべき内容: [具体的な項目]
```

## 🤝 コラボレーション支援

### ステークホルダー別説明生成

```
以下のPRDを、それぞれのステークホルダー向けに説明してください：

[PRDの内容を貼り付け]

以下の3つのバージョンを作成してください：

1. **エンジニア向け** (技術的詳細重視)
2. **デザイナー向け** (UI/UX、ユーザー体験重視)
3. **営業/CS向け** (顧客価値、競合優位性重視)

各バージョンは5文程度で簡潔に。
```

### FAQ自動生成（CS向け）

```
以下のドキュメント群からCS向けのFAQを生成してください：

**PRD**: [内容]
**Tech Spec**: [内容]
**Help Article**: [内容]

以下の形式で10個のQ&Aを生成してください：

**Q: [よくある質問]**
**A**: [回答（平易な言葉で）]
**エスカレーション**: [いつ上位にエスカレートすべきか]
**関連ドキュメント**: [リンク]

カテゴリ分け：
- 一般的な質問（3個）
- 機能の使い方（3個）
- トラブルシューティング（3個）
- 請求・契約（1個）
```

## 🔧 ドキュメントメンテナンス

### 古いドキュメントの識別

```
以下のドキュメントリストを分析し、更新が必要なドキュメントを特定してください：

[全ドキュメントのリストとlast_updated日付を貼り付け]

以下の基準で分類してください：
- 🔴 緊急: 6ヶ月以上更新なし、かつアクティブステータス
- 🟡 要確認: 3-6ヶ月更新なし
- 🟢 問題なし: 3ヶ月以内に更新

出力形式：
## 更新が必要なドキュメント
### 🔴 緊急
- [ドキュメント名] (最終更新: YYYY/MM/DD) - 推奨アクション: [説明]
```

### リンク切れチェック

```
以下のドキュメント内のリンクをチェックしてください：

[ドキュメントの内容を貼り付け]

以下をチェックしてください：
- 相対パスが正しいか
- 参照先のファイルが存在するか（ファイル名の形式から判断）
- プレースホルダー（`(link)`等）が残っていないか

出力形式：
## リンクチェック結果
- ✅ 正常: [リンク]
- ⚠️ 要確認: [リンク] - 理由: [説明]
- ❌ エラー: [リンク] - 理由: [説明]
```

## 🎯 特殊用途プロンプト

### ADR（アーキテクチャ決定）作成支援

```
以下の技術選定についてADRを作成してください：

**問題**: [解決すべき技術的課題]
**候補**: [検討している選択肢を3-5個]

テンプレートに従って作成してください：
- 参照: [プロジェクトパス]/docs/50_Engineering_Tech/52.01_ADR_<Title>.md

以下を含めてください：
- 背景とコンテキスト
- 各候補のPros/Cons（技術的、コスト的、運用的観点）
- 推奨する決定とその理由
- 結果（将来の影響予測）
```

### Post-Mortem（障害報告）作成支援

```
以下の障害情報からPost-Mortemを作成してください：

**発生日時**: [YYYY/MM/DD HH:MM]
**影響**: [影響を受けたユーザー数、サービス停止時間]
**原因**: [初期調査結果]

テンプレートに従って作成してください：
```

## 💡 使い方のTips

### 効果的なプロンプトの書き方

1. **具体的な情報を提供する**: テンプレートだけでなく、プロジェクト固有の情報も含める
2. **出力形式を明示する**: Markdown、箇条書き、表形式など
3. **制約を明確にする**: 文字数、トーン、対象読者
4. **例を示す**: 期待する出力の例を1-2個示すと精度が上がる

### プロンプトのカスタマイズ

このテンプレートはあくまで出発点です。以下のようにカスタマイズしてください：

- プロジェクト固有の用語や制約を追加
- 社内の承認プロセスを反映
- チーム特有のベストプラクティスを組み込む

### AIとの協働のベストプラクティス

- ✅ AIの出力は必ずレビューする
- ✅ 重要な意思決定は人間が行う
- ✅ AIを「ドラフト作成者」として活用し、人間が最終編集
- ❌ AIの出力をそのまま使わない
- ❌ セキュリティ/機密情報をAIに入力しない（プロンプトを一般化する）
