# 50_Engineering_Tech - Index

## カテゴリ概要

技術仕様、アーキテクチャ決定、API仕様、障害報告など、エンジニアリングに関するドキュメント群です。
技術的な意思決定の記録と、システムの設計・運用の知見を蓄積します。

---

## 対象職種

- **主要**: Engineer, TechLead, EM
- **関係者**: PdM, SRE, QA

## 更新頻度

- **Tech Spec**: 機能ごと（実装前に完成）
- **API Spec**: バージョンごと、変更時に即座に更新
- **ADR**: 重要な技術的意思決定ごと
- **Post-Mortem**: 障害発生後1週間以内

## 使い分けガイド

### PRD vs Tech Spec

- **PRD**: **何を**実現するか（ビジネス要件、ユーザー要件）- PdM作成
- **Tech Spec**: **どのように**実装するか（技術的設計）- Engineer作成

### Tech Spec vs ADR

- **Tech Spec**: 個別機能の詳細設計
- **ADR**: 全体アーキテクチャに影響する重要な決定

### Tech Spec vs API Spec

- **Tech Spec**: システム内部の設計（データモデル、ビジネスロジック）
- **API Spec**: 外部向けインターフェース（エンドポイント、レスポンス形式）

## 新規作成ガイド

### Tech Specを作成する場合

1. テンプレート: このカテゴリ内のファイルをテンプレートとして使用
2. 前提条件:
   - PRDが承認済み
   - Design Handoffが完了（UI/UX要件が明確）
3. 必須セクション:
   - アーキテクチャ概要
   - データモデル
   - APIエンドポイント
   - セキュリティ考慮事項
   - パフォーマンス要件
   - テスト戦略
4. レビュープロセス:
   - Engineer作成 → TechLeadレビュー → EMレビュー → 承認 → 実装開始

### ADRを作成する場合

1. テンプレート: このカテゴリ内のファイルをテンプレートとして使用
2. 作成タイミング:
   - 技術スタック選定時（DB、フレームワーク、言語）
   - アーキテクチャ変更時（モノリス→マイクロサービスなど）
   - 重要なライブラリ選定時
3. 必須セクション:
   - 背景・問題提起
   - 検討した選択肢（複数）
   - 決定内容と理由
   - Pros/Cons
4. ステータス管理:
   - Proposed（提案）→ Accepted（承認）→ Deprecated（非推奨）→ Superseded（新ADRで置換）

### API Specを更新する場合

1. テンプレート: このカテゴリ内のファイルをテンプレートとして使用
2. 更新タイミング:
   - 新規エンドポイント追加時
   - 既存エンドポイント変更時（必ずバージョニング）
   - エラーコード追加時
3. 破壊的変更の場合:
   - メジャーバージョンアップ（v1 → v2）
   - 移行期間の設定（v1を6ヶ月間並行運用など）
   - パートナー・顧客への事前通知

### Post-Mortemを作成する場合

1. テンプレート: このカテゴリ内のファイルをテンプレートとして使用
2. 作成タイミング: 障害発生後1週間以内
3. 必須セクション:
   - タイムライン（詳細な時系列）
   - 根本原因（5 Whys分析）
   - 再発防止策（SMART形式のアクションアイテム）
4. レビュー: EM → 全エンジニア → 全社共有
5. 重要: **責任追及ではなく、学びと改善に焦点**

## 技術仕様作成のベストプラクティス

### 良いTech Specの特徴

- **実装可能なレベルの詳細度**: エンジニアが迷わず実装できる
- **セキュリティを最優先**: 脆弱性を事前に考慮
- **パフォーマンスを考慮**: スケーラビリティ、レスポンスタイム
- **テスト戦略が明確**: Unit/Integration/E2Eテストの方針
- **運用を考慮**: モニタリング、ログ、エラーハンドリング

### よくある失敗パターン

- ❌ **Why（なぜ）がない** → PRDへの参照を明記
- ❌ **エラーハンドリングが不明確** → 例外ケースを網羅
- ❌ **パフォーマンス要件がない** → 応答時間、スループットを明記
- ❌ **セキュリティが後付け** → 設計段階で考慮

## ADR作成のベストプラクティス

### 良いADRの特徴

- **複数の選択肢を検討**: 最低3つの選択肢を比較
- **トレードオフが明確**: メリットだけでなくデメリットも正直に記載
- **決定の背景が詳細**: 将来「なぜこの決定をしたのか」が分かる
- **ステータス管理**: 決定後も状況変化に応じてステータス更新

### ADRを書くべき判断基準

- ✅ **書くべき**: 後から変更が困難、影響範囲が大きい
- ❌ **不要**: 局所的、簡単に変更可能

## よくある質問

### Q: Tech SpecとPRDはどちらを先に書きますか？

**A**: PRDが先です。PRDで「何を実現するか」を定義し、Tech Specで「どのように実装するか」を設計します。

### Q: ADRは誰が承認しますか？

**A**: TechLead、EM、CTO。影響範囲が大きい場合はPdMも含めます。

### Q: API仕様はどこまで詳細に書くべきですか？

**A**: OpenAPI (Swagger) 形式で、すべてのエンドポイント、パラメータ、レスポンスを記載。サンプルリクエスト/レスポンスも含めてください。

### Q: Post-Mortemは必ず書きますか？

**A**: 以下の場合は必須です。

- サービス停止（部分的でも）
- データ損失・不整合
- セキュリティインシデント
- ユーザーへの影響が大きい不具合

### Q: Tech Specは途中で変更できますか？

**A**: はい。ただし、大きな設計変更の場合は再度レビューと承認が必要です。変更履歴に記録してください。

## 技術的負債の管理

Tech Specに「Technical Debt」セクションを設けることを推奨します。

```markdown
## Technical Debt

- [ ] パフォーマンス最適化（N+1クエリの解消）
- [ ] テストカバレッジ向上（現状60% → 80%）
- [ ] リファクタリング（肥大化したファイルの分割）
```
