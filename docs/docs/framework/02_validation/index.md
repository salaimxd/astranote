## プロダクト検証とは

プロダクト検証（Product Validation）は、**アイデアや仮説が本当に顧客のニーズを満たすのかを確認するプロセス**です。「作りたいもの」ではなく「顧客が本当に必要としているもの」を作るために、最小限の製品（MVP）で素早く検証を繰り返します。

ディスカバリーフェーズで発見した課題やアイデアを、実際の市場で試し、学びを得ることで、無駄な開発を避け、成功確率を高めることができます。

## リーンスタートアップとは

リーンスタートアップは、**最小限の製品を市場に投入し、顧客からのフィードバックを得ながら、仮説検証を高速で繰り返して事業を成長させる手法**です。

### Build-Measure-Learn（構築-計測-学習）サイクル

リーンスタートアップの中核となる考え方です。

```
1. Build（構築）
   ↓
   MVP（最小実行可能製品）を素早く作る

2. Measure（計測）
   ↓
   顧客の実際の行動データを収集・分析

3. Learn（学習）
   ↓
   データから学び、次の仮説を立てる

（サイクルを繰り返す）
```

**重要なポイント:**

- 短いサイクルでの仮説検証を重視
- 大規模リリース前にMVPでユーザー反応を確認
- 意見ではなく、実際の行動データで検証

## 検証する2つの仮説

リーンスタートアップでは、仮説を大きく2つに分けて考えます。

### 1. 価値仮説（Value Hypothesis）

**問い:** このプロダクトは顧客の課題を解決するのか？

**検証すること:**

- 顧客は本当にこの課題を抱えているか
- 提案するソリューションは課題を解決できるか
- 顧客はこのソリューションに価値を感じるか

**検証方法:**

- ユーザーリサーチ（[User Research](../01_discovery/user_research.md)参照）
- ランディングページでの反応テスト
- プロトタイプを使ったユーザビリティテスト

### 2. 成長仮説（Growth Hypothesis）

**問い:** このサービスはどのように拡大・成長するのか？

**検証すること:**

- 顧客獲得のチャネルは機能するか
- 収益モデルは持続可能か
- スケールできるビジネスモデルか

**検証方法:**

- チャネルテスト（広告、SEO、紹介など）
- 価格設定テスト
- コホート分析

## MVP（Minimum Viable Product）

MVPは、**顧客から学ぶために必要最小限の機能を持った製品**です。Build-Measure-Learnサイクルの起点となります。

### MVPの目的

- ❌ 完璧な製品を作る → ✅ 最速で学ぶ
- ❌ 多くの機能を盛り込む → ✅ 必要最小限の機能で検証
- ❌ 全ての顧客を満足させる → ✅ アーリーアダプターから学ぶ

### MVPの代表的なタイプ

**1. コンシェルジュ型（Concierge MVP）**

- 手動でサービスを提供し、ユーザー行動を観察

**2. ペーパーモック型（Paper Mock MVP）**

- 紙やスライドを使って反応を確かめる

**3. ランディングページ型（Landing Page MVP）**

- LPで価値提案を説明し、登録や問い合わせを計測

**4. オズの魔法使い型（Wizard of Oz MVP）**

- 自動化されているように見せて、裏側は手動で対応

**5. シングル機能型（Single Feature MVP）**

- 1つの核となる機能に絞って実装し、価値仮説を検証

## このセクションで扱うツール

### [リーンキャンバス（Lean Canvas）](./lean_canvas.md)

**目的:** ビジネスモデルを9つの要素で可視化し、仮説を整理

**使うタイミング:**

- アイデアをビジネスモデルとして整理したい時
- チームで事業の全体像を共有したい時
- 仮説を明確にして検証計画を立てたい時

**期待される効果:**

- A4用紙1枚でビジネスモデルを俯瞰
- 分厚い事業計画書より素早く作成・更新できる
- 仮説が明確になり、検証ポイントが見える

**9つの要素:**

1. 顧客セグメント
2. 顧客の課題
3. 独自の価値提案
4. 解決策
5. チャネル
6. 収益の流れ
7. コスト構造
8. 主要指標
9. 競合優位性

### [MVP開発ドキュメント](./mvp.md)

**目的:** MVP の計画、開発、検証を体系的に記録する Docs as Code テンプレート

**使うタイミング:**

- MVPの開発を計画する時
- 検証結果を記録・共有したい時
- チームで開発プロセスを管理したい時

**期待される効果:**

- プロジェクトごとにコピーして使える実践的なテンプレート
- バージョン管理で変更履歴を追跡
- 仮説、検証結果、学びを一元管理
- 意思決定（継続/ピボット/中止）の根拠を明確化

**主要セクション:**

- 仮説定義（価値仮説、成長仮説）
- ターゲットユーザー
- 機能スコープ
- 成功基準とKPI
- 検証計画（MVPキャンバス）
- 開発・リリース計画
- 検証結果の記録
- 学びと意思決定

## 検証のプロセス

```
1. 仮説の整理
   ↓
   リーンキャンバスでビジネスモデルを可視化
   価値仮説と成長仮説を明確化

2. 検証計画の策定
   ↓
   何を検証するか（最もリスクの高い仮説）
   どうやって検証するか（MVP、実験方法）
   成功の基準は何か（KPI、目標値）

3. MVPの構築
   ↓
   最小限の機能で素早く作る
   完璧さより速度を優先

4. 実験の実施
   ↓
   実際のユーザーに使ってもらう
   行動データを収集

5. データの計測・分析
   ↓
   定量データ（数値、行動ログ）
   定性データ（フィードバック、ユーザーレビュー）

6. 学習と意思決定
   ↓
   仮説は正しかったか？
   次に何を検証するか？

   → 継続（Persevere）: 現在の方向性を続ける
   → ピボット（Pivot）: 方向性を転換する
   → 中止（Stop）: このアイデアは諦める

（2〜6を繰り返す）
```

## Discoveryとの関係

プロダクト開発は、DiscoveryとValidationを行き来しながら進みます。

**Discovery（発見）**

- 顧客は誰か？
- どんな課題を抱えているか？
- どんな機会があるか？
- ツール: ペルソナ、共感マップ、HMW

**Validation（検証）**

- そのアイデアは本当に価値があるか？
- ビジネスとして成立するか？
- どうやって成長させるか？
- ツール: リーンキャンバス、MVP、実験

### 典型的な流れ

1. **Discovery**: ペルソナと共感マップでユーザーを深く理解
2. **Discovery**: HMWで課題を機会に転換
3. **Validation**: リーンキャンバスでビジネスモデルを整理
4. **Validation**: MVPで価値仮説を検証
5. **Discovery**: ユーザーフィードバックから新たなインサイトを発見
6. **Validation**: リーンキャンバスを更新し、成長仮説を検証
7. **繰り返し...**

## 検証で大切なこと

**意見ではなく、行動で検証する**

- 「使いたい」という意見 → あてにならない
- 実際に登録する、購入する → 本当のニーズ

**失敗から学ぶ**

- 仮説が外れることは悪いことではない
- 早く失敗することで、大きな損失を防ぐ
- 学びを次のサイクルに活かす

**完璧さより速度**

- 完璧なプロダクトを作るのは時間がかかる
- 最小限のMVPで素早く学ぶ方が価値がある
- 30分でリーンキャンバスを埋める

**顧客と対話し続ける**

- 机上の空論にならないよう、常に顧客の声を聞く
- 定期的にユーザーリサーチを実施
- データだけでなく、質的なフィードバックも重要

**大胆にピボットする**

- 仮説が間違っていたら、方向転換を恐れない
- 執着せず、学びに基づいて柔軟に変更
- ピボットのタイミングを逃さない

## はじめ方

1. **[リーンキャンバス](./lean_canvas.md)を30分で埋める**
   - 完璧である必要はない
   - すべての要素に仮説を記入
   - チームで共有

2. **最もリスクの高い仮説を特定**
   - 顧客セグメントは正しいか？
   - 課題は本当に存在するか？
   - ソリューションは受け入れられるか？

3. **検証計画を立てる**
   - どのMVPタイプを使うか
   - 何を計測するか
   - 成功の基準は何か

4. **素早く実験する**
   - MVPを作成
   - 実際のユーザーでテスト
   - データを収集

5. **学びをリーンキャンバスに反映**
   - 仮説を更新
   - 次の検証に進む
   - 検証が完了したら[Refinement（深化）](../03_refinement/)セクションへ
