## 7. 検証計画（MVPキャンバス）

### 7.1 仮説

**最も重要な仮説:**

> ラノベ特化AI翻訳（プロンプトエンジニアリングによるキャラクター名辞書・文体設定・コンテキスト保持）が、既存のファンサブ翻訳と同等以上の品質を実現し、海外ラノベ読者の70%以上が「十分に読める」と評価する。

### 7.2 リスク

**想定されるリスク:**

| リスク | 影響度 | 発生確率 | 対応策 |
|--------|--------|----------|--------|
| 1. AI翻訳品質がラノベ読者の没入感を満たせない | 最高 | 中 | 人間翻訳者とのハイブリッドモデルへピボット |
| 2. 作品ラインナップ不足（テスター作家の作品が少ない） | 高 | 中 | テスター数を50名に増員、既存人気作品の許諾取得 |
| 3. 技術的スケーラビリティ（API呼び出し制限、レスポンス遅延） | 中 | 低 | APIプロバイダの分散（Claude + GPT-4）、キャッシュ戦略 |
| 4. テスター離脱（β版期間中の離脱率が高い） | 中 | 中 | 週次フォローアップ、謝礼の段階支給 |
| 5. 競合の先行リリース（Webnovel等が類似機能をリリース） | 低 | 低 | ラノベ特化の深さで差別化、速度よりも品質を優先 |

**対応策:**

- リスク1（翻訳品質不足）: Week 3-4でプロンプトの集中的なチューニングを実施。品質が目標に達しない場合、人間翻訳者によるポストエディット（AI翻訳→人間校正）のハイブリッドモデルを検討
- リスク2（ラインナップ不足）: 事前に5作品以上のテスト用コンテンツを確保。既存のCC-BY作品やテスター自身の作品を活用
- リスク3（スケーラビリティ）: 翻訳結果のキャッシュ、バッチ処理の導入。APIの並列呼び出しでレスポンス改善

### 7.3 検証方法

**実施内容:**

- [x] A/Bテスト: 同一作品のAI翻訳版 vs 既存ファンサブ翻訳版をブラインドで読者に提示し、品質を比較評価
- [x] NPS調査: 「astranoteを友人に薦めますか？」のNPSスコアを作家・読者それぞれに実施
- [x] ユーザーインタビュー: β版参加者から作家10名・読者20名を選定し、30分の1on1インタビューを実施
- [x] 行動ログ分析: Mixpanelで翻訳フロー・読書フローのファネルを分析
- [x] 翻訳品質の専門評価: ラノベ翻訳経験のある翻訳者2名による専門評価

**期間:** 2026-05-12 〜 2026-06-09（β版リリース後4週間）

**必要なリソース:**

- Mixpanel（行動ログ分析）
- Hotjar（ヒートマップ・セッション録画）
- Googleフォーム（翻訳品質アンケート・NPSアンケート）
- Zoom/Google Meet（ユーザーインタビュー）
- 翻訳者2名（専門評価用、外部パートナー）

### 7.4 計測するデータ

**定量データ:**

- 翻訳品質スコア（5段階評価の平均）
- A/Bテストの選好率（AI翻訳 vs ファンサブ翻訳）
- ファネル分析: 登録→初回翻訳→公開→読者閲覧の各ステップの転換率
- 読書行動: 1セッションあたりの読了章数、平均滞在時間、離脱章
- リテンション: Day 1, Day 7, Day 14, Day 28のリテンション率
- NPS: 作家NPS、読者NPS

**定性データ:**

- ユーザーインタビューの記録（音声+テキスト書き起こし）
- 翻訳品質に関する自由記述フィードバック
- 作家の翻訳修正ログ（どの箇所をどのように修正したか）
- X（Twitter）でのastranote言及のセンチメント分析

**計測ツール:**

- アナリティクス: Mixpanel（ファネル分析、リテンション、イベントトラッキング）
- ヒートマップ: Hotjar（読書ページのスクロール率、クリック分析）
- フィードバック収集: Googleフォーム（アンケート）、Notion（インタビュー記録）
- その他: カスタム翻訳品質評価フォーム（5段階評価+自由記述、作品ごと・章ごとに評価）

## 8. 開発計画

### 8.1 技術スタック

> **アーキテクチャ方針の変更:** コスト最適化の観点から、当初計画のVercel + Supabase構成を見直し、**Cloudflareエコシステムに統一**する設計に変更した。これによりMVP段階のインフラ固定費をほぼ$0に抑え、スケーラビリティも確保する。

**設計方針:**

- **コストファースト**: 無料枠・低コストサービスを最大活用し、MVP段階での固定費を最小化
- **Cloudflare統一**: フロントエンド・バックエンド・データベース・ストレージを一つのプラットフォームで完結
- **段階的スケール**: 無料枠で開始し、トラフィック増加に応じて有料プランへ移行

**フロントエンド:**

| カテゴリ | 技術 | 選定理由 |
|---------|------|----------|
| フレームワーク | Astro (SSG/SSR) | 軽量、Islands Architectureで必要な部分のみJS。読書ビューア等の静的ページはJS 0KB |
| インタラクティブUI | React (Islands) | エディタ・ダッシュボード等、インタラクティブが必要な部分のみReactコンポーネントをロード |
| 言語 | TypeScript | 型安全、フロント・バックエンド統一 |
| スタイリング | Tailwind CSS + shadcn/ui | 軽量、カスタマイズ性 |
| 状態管理 | nanostores | 超軽量、フレームワーク非依存 |
| ホスティング | Cloudflare Pages | 無料（無制限リクエスト、500ビルド/月） |

**バックエンド:**

| カテゴリ | 技術 | 選定理由 |
|---------|------|----------|
| ランタイム | Cloudflare Workers | エッジ実行で低遅延、無料100Kリクエスト/日 |
| フレームワーク | Hono | Workers最適化、超軽量、Web Standards準拠 |
| ORM | Drizzle ORM | D1対応、TypeScript、軽量 |
| 認証 | Lucia Auth | Workers対応、D1連携、OAuth（Google / GitHub / Discord） |
| バリデーション | Zod | 軽量、TypeScript統合 |

**データストア:**

| カテゴリ | 技術 | 無料枠 | 用途 |
|---------|------|--------|------|
| メインDB | Cloudflare D1 (SQLite) | 5GB、5M reads/日、100K writes/日 | ユーザー、作品、翻訳結果、評価データ |
| キャッシュ | Cloudflare KV | 100K reads/日 | セッション、ランキング、翻訳キャッシュ |
| ストレージ | Cloudflare R2 | 10GB、Egress無料 | カバー画像、アセット |
| 全文検索 | D1 FTS5 (SQLite組み込み) | 無料 | 作品検索 |

**AI翻訳（コスト最適化 多層構成）:**

| モデル | 入力コスト | 出力コスト | 1章あたり（3,000文字） | 用途 |
|--------|-----------|-----------|----------------------|------|
| **Gemini 2.0 Flash** | $0.10/1M tokens | $0.40/1M tokens | **$0.002** | **メイン翻訳（最安）** |
| Claude 3.5 Haiku | $0.25/1M tokens | $1.25/1M tokens | $0.006 | 高品質が必要な場合 |
| DeepL API Free | 無料 | 無料 | $0 | バックアップ（500K文字/月まで） |

> **POCとの関係:** POC（翻訳品質スパイク）では複数モデルの翻訳品質を比較評価する。POC結果に基づき、メイン翻訳モデルを最終決定する。Gemini 2.0 Flashはコスト面で最適だが、品質がClaude API等に劣る場合はモデル比率を調整する。

**インフラ:** Cloudflare (Pages + Workers + D1 + KV + R2) — 統一プラットフォーム

**ツール:**

- Mixpanel: ユーザー行動分析（無料プラン、月1,000 MTU）
- Hotjar: ヒートマップ・セッション録画
- Sentry: エラー追跡（Developer Free）
- Cloudflare Analytics: Webトラフィック解析（無料）
- GitHub: ソースコード管理
- GitHub Actions: CI/CD
- Stripe: 将来の決済用（MVP後に実装）

**選定理由:**

個人〜少人数チームでのMVP開発に最適化し、**コスト最小化**と**開発速度**を両立する技術スタックを選定した。

| 選定基準 | 選定理由 |
|----------|----------|
| コスト最適化 | Cloudflareエコシステムに統一することで、MVP段階のインフラ固定費をほぼ$0に削減。Vercel Pro ($20/月) + Supabase Pro ($25/月) の組み合わせと比較して90%以上のコスト削減 |
| 開発速度 | Hono + Drizzle ORM + Lucia Auth の組み合わせにより、認証・DB・APIをシンプルに構築可能。Cloudflare D1はマイグレーション対応で開発体験も良好 |
| パフォーマンス | Astro Islands アーキテクチャにより、読書ビューア等の静的ページはJS 0KBで超高速表示。インタラクティブ部分（エディタ、ダッシュボード）のみReactをロード。Cloudflare Workersのエッジ実行で低遅延API応答 |
| スケーラビリティ | Cloudflare Workers + D1はトラフィック増加に対して自動スケール。Phase 1（無料枠）→ Phase 2（Workers Paid $5/月）→ Phase 3（D1→Turso移行）と段階的に拡張可能 |
| AI翻訳コスト | Gemini 2.0 Flashをメイン翻訳に採用し、翻訳コストを1章あたり$0.002に抑制。Claude Sonnet ($0.07/章) 比で97%削減 |
| TypeScript統一 | フロントエンド（Astro + React）・バックエンド（Hono Workers）をTypeScriptで統一し、コードの共有と保守性を向上 |

**フェーズ別コスト見積もり:**

| フェーズ | 想定ユーザー数 | インフラ月額 | 翻訳API月額 | 合計 |
|---------|---------------|-------------|-------------|------|
| Phase 1: MVP | ~1,000 | **~$0**（全て無料枠内） | ~$3（Gemini Flash 50章/日） | **~$4/月** |
| Phase 2: 成長期 | ~10,000 | ~$15（Workers Paid + D1 Paid） | ~$50（Gemini + Claude Haiku） | **~$70/月** |
| Phase 3: スケール期 | ~100,000 | ~$90（D1→Turso移行含む） | ~$200（Gemini + Claude） | **~$300/月** |

### 8.4 技術的な制約・前提

**制約:**

- Cloudflare Workers無料枠のリクエスト制限（100,000リクエスト/日）
- Cloudflare Workers無料枠のCPU時間制限（10ms/リクエスト。翻訳API呼び出しの待機時間は含まない）
- Cloudflare D1無料枠の書き込み制限（100K writes/日。MVP期間中は十分）
- AI翻訳APIのレート制限（Gemini API: 1分あたり1,500リクエスト、Claude API: 1分あたり60リクエスト）
- 1章あたりの翻訳レスポンスタイム目標10秒以内（5,000文字程度を想定）
- 個人開発のため、24時間対応のサポート体制は構築しない

**前提:**

- AI翻訳APIの料金体系がMVP期間中に大幅に変更されないこと
- Cloudflareの無料枠がMVP期間中に大幅に縮小されないこと
- なろう/カクヨムの利用規約が、外部サービスへのテキストコピーを禁止していないこと
- β版テスターが十分な数（作家30名・読者100名）集まること
- ラノベ特化のプロンプトエンジニアリングにより、汎用翻訳を超える品質が実現可能であること

### 8.5 技術的リスクと対策

MVP開発における技術的リスクと、各リスクへの対策を以下に定義する。これらのリスクは、翻訳品質POCの結果や開発進行状況に応じて継続的に更新する。

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|----------|------|
| **リスク1: AI翻訳品質の不確実性**（ラノベ特有文体への対応不足） | 最高 | 中 | **POCで事前検証**（MVP本開発前に2週間のスパイクを実施し、4.0/5.0以上を確認してから着手。POCがNo-Goの場合は人間翻訳者+AIのハイブリッドモデルへピボット） |
| **リスク2: APIコスト変動**（翻訳コストが想定を超える） | 高 | 中 | **文字数上限の設定**（MVP期間は月間翻訳文字数に上限を設定。超過分は作家に通知）または**パススルー設計**（翻訳コストを作家側に実費請求するモデルへの変更を検討） |
| **リスク3: なろう/カクヨム利用規約リスク** | 高 | 低〜中 | **法務レビューの実施**（MVP開発開始前に専門家によるレビュー実施）＋**作家自身のコピペフロー設計**（自動スクレイピングは行わず、作家が自身でテキストをペーストするフローのみ提供。スクレイピング機能は実装しない） |
| **リスク4: 縦書きCSS対応の複雑さ** | 中 | 高 | **MVP期間は横書きのみ**に限定。縦書きは日本の読者体験には重要だが、英語読者向けには横書きが標準であるため、MVP段階での影響は限定的。縦書き対応はβ版以降に計画する |
| **リスク5: レスポンスタイムが目標を超える** | 中 | 中 | **バッチ翻訳＋キャッシュ戦略**を導入。翻訳リクエスト送信後にバックグラウンドで処理し、完了時に通知する非同期フローを採用。リアルタイム翻訳（10秒以内）が困難な場合は、「翻訳中」の待機UXを設計する |
| **リスク6: β版テスターの離脱** | 中 | 中 | 週次のフォローアップ（プログレスメール+Discord通知）と、段階的なテスター謝礼支給（翻訳完了数に応じた報酬）で継続率を維持 |

### 8.7 品質基準

**最低限の品質基準（横書き読書ビューア前提）:**

- [x] 翻訳レスポンスタイム: 5,000文字あたり10秒以内（非同期フロー採用時は「翻訳完了通知」方式でも可）
- [x] サービス稼働率: 99.5%以上（計画メンテナンス除く）
- [x] アクセシビリティ: WCAG 2.1 AA準拠（読書ビューアのフォントサイズ・コントラスト）
- [x] モバイルレスポンシブ: iOS Safari、Android Chrome での読書体験を保証
- [x] データ保護: 作家の原稿データの暗号化保存、HTTPS通信の強制

**テスト方針:**

- ユニットテスト: 翻訳エンジンのコア機能（文字数カウント、章分割、辞書適用）
- E2Eテスト: 主要ユーザーフロー（登録→執筆→翻訳→公開→読了）のPlaywrightテスト
- 翻訳品質テスト: 10作品×3章=30パターンの翻訳結果を人間評価で品質チェック
- 負荷テスト: 同時翻訳リクエスト30件でのレスポンスタイム検証

### 9.3 ロールバック計画

**緊急時の対応:**

Cloudflare Pagesのインスタントロールバック機能を活用し、前回の安定版に即座に切り戻す。バックエンド（Workers）は `wrangler rollback` コマンドで前バージョンに復旧する。翻訳APIの障害時は、多層フォールバック設計（Gemini 2.0 Flash → Claude 3.5 Haiku → DeepL API Free）で自動切り替えし、複数API同時障害時にも翻訳サービスを継続できる仕組みを実装する。

> **フォールバック設計の補足:** メイン翻訳（Gemini Flash）の障害時はClaude Haikuに自動フォールバックし、両方が利用不可の場合はDeepL API Freeに切り替える。DeepL Freeはラノベ特化プロンプトが適用されないため、フォールバック時の翻訳品質は主系より低下する可能性があることをユーザーへ通知する。

**ロールバック条件:**

- サービス稼働率が95%を下回った場合（24時間以内）
- 翻訳APIの連続エラー率が10%を超えた場合
- ユーザーデータの損失・漏洩が確認された場合（即時ロールバック+サービス停止）
