## 1. 背景と目的

### 1.1 背景

**プロジェクトの背景:**

Discoveryフェーズにおいて、以下の6つのドキュメントを完成させ、Web小説のグローバル展開における課題を深く理解した。

1. **ユーザーリサーチ** — 作家・読者双方の生の声を収集し、翻訳コスト・品質・プラットフォームの課題を特定
2. **競合分析** — Syosetu/カクヨム/Webnovel/DeepL等の4社を分析し、「ラノベ特化AI翻訳×作家還元」の空白市場を発見
3. **ペルソナ** — 山田太郎（作家・プライマリ）、Emily Chen（海外読者・セカンダリ）、佐藤美咲（プロ作家・セカンダリ）の3名を定義
4. **共感マップ** — 各ペルソナの思考・感情・行動パターンを可視化し、痛みと期待を明確化
5. **ジャーニーマップ** — 「執筆→翻訳→公開→読者獲得→収益化」の全プロセスにおけるペインポイントをマッピング
6. **HMW（How Might We）** — 「どうすれば翻訳の壁を壊せるか」「どうすれば作家に正当な収益を届けられるか」「どうすれば読者の没入体験を実現できるか」の3つの問いを設定

**解決したい課題:**

| 課題 | 内容 | 影響 |
|------|------|------|
| 翻訳の壁 | 日本語Web小説の英語翻訳コストが1作品50〜100万円、期間6ヶ月以上 | 99%以上の作品が海外読者に届かない |
| 収益構造の歪み | 既存プラットフォームの印税率5〜15%、投げ銭手数料50%以上 | 作家の創作モチベーション低下 |
| プラットフォーム老朽化 | なろう/カクヨムのUI/UXが2010年代のまま、海外読者向け機能なし | グローバル展開の障壁 |

### 1.2 MVPの目的

**このMVPで検証したいこと:**
- [x] ラノベ特化AI翻訳が、ファンサブ翻訳と同等以上の品質（4.0/5.0以上）を実現できるか
- [x] なろう/カクヨムの作家が月¥1,000の翻訳サービスに支払意思を持つか
- [x] 海外ラノベ読者がAI翻訳作品を「十分に読める」と感じ、継続的に読むか

**期待される学び:**

- AI翻訳のどの側面（語彙の正確さ、文体の再現性、キャラクターの一人称、固有名詞の一貫性）が読者の没入感に最も影響するかを理解する
- 作家が翻訳サービスに求める最低限の機能セットを特定する
- 海外読者の作品発見から継続読書に至るユーザージャーニーを実データで検証する

## 2. 仮説定義

### 2.1 価値仮説（Value Hypothesis）

**誰が:** 山田太郎タイプ（なろう/カクヨムで連載中、累計10万PV以上、海外展開に興味があるがコスト障壁で諦めている作家）

**何に困っていて:** 翻訳コスト50〜100万円・期間6ヶ月以上の障壁により、自分の作品を海外読者に届けることができない。既存の機械翻訳（Google翻訳・DeepL）ではラノベ特有の文体（一人称の使い分け、異世界固有名詞、ルビ表現）が正しく翻訳されず実用に耐えない。

**何を提供すれば:** ラノベ特化AI翻訳エンジン（月¥1,000で無制限翻訳、ワンクリックで日→英翻訳、キャラクター名辞書・文体設定対応）

**どうなるか:** 作家が月¥1,000で自分の作品を即座に英語圏の読者に届けられるようになり、「自分の作品が世界で読まれている」という体験を得る。Emily Chenタイプの海外読者が「borderline unreadable」なファンサブ翻訳から「没入できる品質」の翻訳を得られるようになる。

**検証方法:**
- A/Bテスト: AI翻訳 vs 既存ファンサブ翻訳の品質比較（読者によるブラインド評価）
- 翻訳品質スコア: 5段階評価アンケート（目標4.0/5.0以上）
- 作家の初回翻訳完了率: 登録→翻訳実行までの完了率（目標40%以上）

### 2.2 成長仮説（Growth Hypothesis）

**どのチャネルで:** X（Twitter）の日本語Web小説作家コミュニティ、Reddit r/LightNovels・r/noveltranslations、Discord翻訳コミュニティ

**どのように顧客を獲得し:**
- 作家側: X（Twitter）でのβテスター募集、なろう/カクヨムの上位ランキング作家への直接DM、作家コミュニティでの口コミ
- 読者側: Reddit/Discordでの翻訳作品公開、英語圏ラノベレビューサイトとの連携

**どのように収益化するか:**
- 作家: 月額¥1,000のサブスクリプション（翻訳枚数無制限）
- 読者: 月額$5のサブスクリプション（全作品読み放題）
- 投げ銭: 30%プラットフォーム手数料、70%作家還元

**検証方法:**
- LP登録率: β版LP訪問→メール登録の転換率（目標5%以上）
- 作家アンケート: 月¥1,000の支払意思（目標60%以上が「払う」と回答）
- 読者無料→有料転換率: 無料期間後のサブスク加入率（目標5%以上）

### 2.3 最もリスクの高い仮説

**このMVPで最優先で検証すべき仮説:**

> ラノベ特化AI翻訳が、ファンサブ翻訳と同等以上の品質を実現し、海外読者の70%以上が「十分に読める」と評価する

**なぜこれが最もリスクが高いか:**

この仮説はastranoteのプロダクト価値の根幹を成す。AI翻訳の品質が不十分であれば、作家は翻訳結果を公開したがらず（供給サイドの崩壊）、読者は離脱する（需要サイドの崩壊）。価格や機能の問題はピボット可能だが、翻訳品質の問題はプロダクトの存在意義そのものに関わる。リーンキャンバスで特定した3つの仮説（AI翻訳品質、作家の支払意思、読者のサブスク加入）のうち、他の2つは価格調整やモデル変更で対応可能だが、翻訳品質の問題は技術的な根本課題であり、後からの修正コストが最も高い。

## 3. ターゲットユーザー

### 3.1 アーリーアダプター

**具体的なペルソナ:** 山田太郎タイプ — なろう/カクヨムで連載中のWeb小説作家

**属性:**
- 年齢・性別: 25〜35歳、男性が中心（Web小説作家の主要層）
- 職業: ITエンジニア、クリエイター等（本業+副業で執筆）
- 居住地: 日本国内（首都圏中心）
- デジタルリテラシー: 高い（日常的にWebサービスを使いこなす、新しいツールへの抵抗感が低い）

**行動特性:**
- なろう/カクヨムで累計10万PV以上、定期連載中（週1〜3回更新）
- X（Twitter）で作家活動を発信、フォロワー1K〜5K
- 海外からのコメントやファンレターを受け取った経験があり、海外展開に関心がある
- 翻訳コストの高さから海外展開を諦めていた（50〜100万円は副業収入では捻出困難）
- 新しいWebサービスやAIツールに積極的に試す「イノベーター」気質

**獲得方法:**
- X（Twitter）でのβテスター募集ツイート（ハッシュタグ #小説家になろう #カクヨム #Web小説 を活用）
- なろう/カクヨムの週間ランキング上位作家へのDM（直接アプローチ）
- Web小説関連のDiscordサーバーでの告知
- 既存の作家コミュニティ（エブリスタ作家の会等）への参加・紹介

**リクルーティング計画:**
- 目標人数: 作家30名（α版5名→β版30名）、読者100名（β版から参加）
- 期間: 開発期間中の6週目までにα版テスター確保、8週目までにβ版テスター確保
- 方法:
  - Week 1-2: ターゲット作家リスト100名作成、DM送付開始
  - Week 3-4: 応募フォーム設置、スクリーニング（作品PV数、連載頻度、海外展開意欲で選定）
  - Week 5-6: α版テスター5名確定、β版候補30名にウェイティングリスト登録依頼
  - Week 7-8: 読者募集（Reddit r/LightNovels、Discord翻訳コミュニティへの投稿）

## 4. MVPタイプの選択

### 4.1 選択したMVPタイプ

**[ ] コンシェルジュ型** - 手動でサービスを提供
**[ ] ペーパーモック型** - 紙やスライドを使って反応を確かめる
**[ ] ランディングページ型** - LPで需要確認
**[ ] オズの魔法使い型** - 自動化に見せて手動対応
**[x] シングル機能型** - 1つの機能に絞った実装
**[ ] その他:**

### 4.2 選択理由

シングル機能型MVPを選択した理由は以下の3点である。

1. **翻訳品質は体験しなければ評価できない**: astranoteの最重要仮説は「AI翻訳品質がラノベ読者の没入感を満たせるか」である。これはランディングページやペーパーモックでは検証不可能であり、実際に動く翻訳エンジンを読者に体験させる必要がある。
2. **コンシェルジュ型では翻訳品質のスケーラビリティを検証できない**: 人間翻訳者による手動対応では、AI翻訳特有の品質課題（文脈理解、固有名詞の一貫性等）が見えない。
3. **プロンプトエンジニアリングの反復検証が必要**: ラノベ特化の翻訳品質を実現するには、プロンプトの調整→翻訳結果の評価→改善のBuild-Measure-Learnサイクルを高速で回す必要があり、動くプロダクトが不可欠。

### 4.3 期間と予算

**開発期間:** 8週間（2026-02-17 〜 2026-04-13）

**予算:** 約500,000円

## 5. 機能スコープ

### 5.1 MVPに含める機能（Must Have）

| 優先度 | 機能名 | 説明 | 検証する仮説 | 工数見積 |
|--------|--------|------|--------------|----------|
| P0 | ラノベ特化AI翻訳（日→英） | Claude API / GPT-4 APIを活用し、ラノベ特有の文体（一人称、固有名詞、ルビ）に対応した日英翻訳。キャラクター名辞書、文体設定（硬め/柔らかめ）をサポート | 仮説1: AI翻訳品質 | 3週間 |
| P0 | 基本的な執筆エディタ（Markdown対応） | Markdownベースのシンプルなエディタ。章・エピソード管理、自動保存、プレビュー機能。なろう/カクヨムからのインポート機能（テキスト貼り付け） | 仮説2: 作家の利用体験 | 2週間 |
| P0 | 翻訳結果の公開機能 | 翻訳された作品をURL共有で公開可能。作品ページ（タイトル、あらすじ、章一覧）、読書ビューア（縦スクロール、フォントサイズ調整） | 仮説3: 読者の読書体験 | 1.5週間 |
| P0 | ユーザー認証 | メールアドレス/Googleアカウントでのサインアップ・ログイン。作家/読者のロール分離 | 全仮説の前提 | 0.5週間 |

### 5.2 MVPに含めない機能（Will Not Have）

**意図的に含めない機能:**
- 投げ銭/決済機能 — Stripe連携、作家への送金フロー
- マルチプラットフォーム同時投稿 — なろう/カクヨム/Webnovelへの自動投稿
- 読者コミュニティ機能 — コメント、フォロー、お気に入り、ランキング
- 多言語翻訳（英語以外） — 中国語、韓国語、スペイン語等への翻訳
- 高度なエディタ機能 — WYSIWYG、画像挿入、共同執筆
- 作家ダッシュボード — PV分析、読者属性分析、収益レポート
- SEO/ASO最適化 — 検索エンジン向けメタデータ最適化

**理由:**

MVPの目的は「AI翻訳品質の検証」に集中すること。決済・コミュニティ・分析等の機能は翻訳品質の仮説が検証された後に追加する。これらの機能をMVPに含めると、開発期間が3〜4倍に膨らみ、検証の焦点がぼやける。リーンキャンバスで定義した損益分岐点（ライター400名+読者2,000名）の達成は正式リリース後の目標であり、MVPフェーズでは追求しない。

### 5.3 次のイテレーションで検討する機能

- 投げ銭機能（Stripe連携、70%作家還元） — 仮説3（読者のサブスク加入）の検証後
- 読者ダッシュボード（PV、フォロワー数、読了率の可視化） — β版のデータを基に設計
- コミュニティ機能（コメント、フォロー） — 読者のエンゲージメント指標を見てから判断
- なろう/カクヨムからのワンクリックインポート — 作家のオンボーディング改善用
- 多言語翻訳（中国語、韓国語） — 英語翻訳の品質が確立された後に展開

## 6. 成功基準とKPI

### 6.1 定量指標

| 指標 | 現状 | 目標値 | 計測方法 | 備考 |
|------|------|--------|----------|------|
| 翻訳品質スコア | N/A（新規） | 4.0/5.0以上 | 読者アンケート（5段階評価） | 最重要指標 |
| 初回翻訳完了率 | N/A | 40%以上 | Mixpanel（登録→翻訳実行のファネル） | 作家のアクティベーション |
| 3話完読率 | N/A | 30%以上 | Mixpanel（読者の読了トラッキング） | 読者のエンゲージメント |
| 7日リテンション | N/A | 25%以上 | Mixpanel（作家のリテンション） | 継続利用の意思 |
| LP登録率 | N/A | 5%以上 | Google Analytics（LP→登録フォーム） | 需要の確認 |

**AARRR指標:**
- **Acquisition（獲得）:**
  - 指標: β版LP訪問→メール登録の転換率
  - 目標: 5%以上（LP訪問者のうち）

- **Activation（活性化）:**
  - 指標: 登録→初回翻訳完了の転換率（作家）、登録→3話完読の転換率（読者）
  - 目標: 作家40%以上、読者30%以上

- **Retention（定着）:**
  - 指標: 7日後の再訪率（作家）、翌週の再読率（読者）
  - 目標: 作家25%以上、読者20%以上

- **Revenue（収益）:**
  - 指標: 「月¥1,000を払う意思がある」と回答した作家の割合（アンケート）
  - 目標: 60%以上（MVPフェーズでは実際の課金は行わない）

- **Referral（紹介）:**
  - 指標: 翻訳作品のURLシェア数、X（Twitter）でのastranote言及数
  - 目標: β版参加者の20%以上がSNSでシェア

### 6.2 定性指標

**観察したいユーザー行動:**
- 作家が翻訳結果を見た瞬間の反応（「おお！」「すごい！」等の感嘆の声があるか）
- 作家が翻訳結果をそのまま公開するか、それとも手動で修正してから公開するか（修正箇所の特定）
- 読者がAI翻訳作品を読み始めてから離脱するまでの読書パターン（どの章で離脱するか）
- 読者がファンサブ翻訳とAI翻訳を比較した時の具体的なフィードバック
- 作家同士でastranoteについて会話しているか（自然発生的な口コミ）

### 6.3 成功/失敗の判断基準

**成功と判断する条件:**
- 翻訳品質スコアが4.0/5.0以上（読者の70%以上が「十分に読める」と評価）
- 初回翻訳完了率が40%以上（作家のアクティベーション）
- 3話完読率が30%以上（読者のエンゲージメント）
- 作家の60%以上が「月¥1,000を払う」と回答

**失敗と判断する条件:**
- 翻訳品質スコアが3.0/5.0未満（読者の過半数が「読めない」と評価）

- 3話完読率が10%未満（読者が離脱する）
- 作家の支払意思が30%未満

**判断のタイミング:** β版開始4週間後（2026-05-11を予定）

## 7. 検証計画（MVPキャンバス）

### 7.1 仮説

**最も重要な仮説:**

> ラノベ特化AI翻訳（プロンプトエンジニアリングによるキャラクター名辞書・文体設定・コンテキスト保持）が、既存のファンサブ翻訳と同等以上の品質を実現し、海外ラノベ読者の70%以上が「十分に読める」と評価する。

### 7.2 リスク

**想定されるリスク:**

| リスク | 影響度 | 発生確率 | 対応策 |
|--------|--------|----------|--------|
| 1. AI翻訳品質がラノベ読者の没入感を満たせない | 最高 | 中 | 人間翻訳者とのハイブリッドモデルへピボット |
| 2. 作品ラインナップ不足（テスター作家の作品が少ない） | 高 | 中 | テスター数を50名に増員、既存人気作品の許諾取得 |
| 3. 技術的スケーラビリティ（API呼び出し制限、レスポンス遅延） | 中 | 低 | APIプロバイダの分散（Claude + GPT-4）、キャッシュ戦略 |
| 4. テスター離脱（β版期間中の離脱率が高い） | 中 | 中 | 週次フォローアップ、謝礼の段階支給 |
| 5. 競合の先行リリース（Webnovel等が類似機能をリリース） | 低 | 低 | ラノベ特化の深さで差別化、速度よりも品質を優先 |

**対応策:**
- リスク1（翻訳品質不足）: Week 3-4でプロンプトの集中的なチューニングを実施。品質が目標に達しない場合、人間翻訳者によるポストエディット（AI翻訳→人間校正）のハイブリッドモデルを検討
- リスク2（ラインナップ不足）: 事前に5作品以上のテスト用コンテンツを確保。既存のCC-BY作品やテスター自身の作品を活用
- リスク3（スケーラビリティ）: 翻訳結果のキャッシュ、バッチ処理の導入。APIの並列呼び出しでレスポンス改善

## 8. 開発計画

### 8.1 技術スタック

**フロントエンド:** Next.js 15 (App Router) + TailwindCSS + shadcn/ui

**バックエンド:** Supabase (Auth / PostgreSQL / Storage / Edge Functions)

**AI翻訳:** Claude API (Anthropic) / GPT-4 API (OpenAI)（ラノベコーパスでのプロンプトエンジニアリング）

**インフラ:** Vercel (フロントエンド・Edge Functions) + Supabase (BaaS)

**ツール:**
- Mixpanel: ユーザー行動分析
- Hotjar: ヒートマップ・セッション録画
- Stripe: 将来の決済用（MVP後に実装）
- GitHub: ソースコード管理
- GitHub Actions: CI/CD

**選定理由:**

個人〜少人数チームでのMVP開発に最適化した技術スタックを選定した。

| 選定基準 | 選定理由 |
|----------|----------|
| 開発速度 | Next.js + Supabase の組み合わせにより、認証・DB・ストレージをほぼゼロコンフィグで利用可能。フルスクラッチ比で開発期間を50%短縮 |
| 運用コスト | サーバーレスアーキテクチャにより、MVP段階での固定サーバー費用がほぼゼロ。従量課金モデルでスモールスタート可能 |
| スケーラビリティ | Vercel + Supabase はトラフィック増加に対して自動スケール。MVP→正式版への移行時にアーキテクチャ変更不要 |
| AI翻訳品質 | Claude API / GPT-4 API は日本語→英語の翻訳品質が高い。プロンプトエンジニアリングにより、ラノベ特化のカスタマイズが可能 |
| TypeScript統一 | フロントエンド・バックエンド（Edge Functions）をTypeScriptで統一し、コードの共有と保守性を向上 |

### 8.4 技術的な制約・前提

**制約:**
- AI翻訳APIのレート制限（Claude API: 1分あたり60リクエスト、GPT-4 API: 1分あたり500リクエスト）
- 1章あたりの翻訳レスポンスタイム目標10秒以内（5,000文字程度を想定）
- Supabase無料プランのDB容量500MB（MVP期間中は十分）
- 個人開発のため、24時間対応のサポート体制は構築しない

**前提:**
- AI翻訳APIの料金体系がMVP期間中に大幅に変更されないこと
- なろう/カクヨムの利用規約が、外部サービスへのテキストコピーを禁止していないこと
- β版テスターが十分な数（作家30名・読者100名）集まること
- ラノベ特化のプロンプトエンジニアリングにより、汎用翻訳を超える品質が実現可能であること

### 8.5 品質基準

**最低限の品質基準:**
- [x] 翻訳レスポンスタイム: 5,000文字あたり10秒以内
- [x] サービス稼働率: 99.5%以上（計画メンテナンス除く）
- [x] アクセシビリティ: WCAG 2.1 AA準拠（読書ビューアのフォントサイズ・コントラスト）
- [x] モバイルレスポンシブ: iOS Safari、Android Chrome での読書体験を保証
- [x] データ保護: 作家の原稿データの暗号化保存、HTTPS通信の強制

**テスト方針:**
- ユニットテスト: 翻訳エンジンのコア機能（文字数カウント、章分割、辞書適用）
- E2Eテスト: 主要ユーザーフロー（登録→執筆→翻訳→公開→読了）のPlaywrightテスト
- 翻訳品質テスト: 10作品×3章=30パターンの翻訳結果を人間評価で品質チェック
- 負荷テスト: 同時翻訳リクエスト30件でのレスポンスタイム検証

### B. 翻訳品質評価基準

| スコア | 評価 | 基準 |
|--------|------|------|
| 5 | 優秀 | プロの翻訳者と遜色ない品質。キャラクターの口調、世界観の雰囲気が忠実に再現されている |
| 4 | 良好 | 多少の不自然さはあるが、没入感を損なわずに読み通せる。ファンサブ翻訳以上の品質 |
| 3 | 普通 | 意味は通じるが、文体の不自然さが目立つ。読了はできるが没入感に欠ける |
| 2 | 不足 | 誤訳や不自然な表現が多く、読み進めるのに苦労する。ファンサブ翻訳以下の品質 |
| 1 | 不可 | 意味が通じない箇所が多く、読了が困難。実用に耐えない |

### C. プロンプトエンジニアリング方針

MVP期間中のAI翻訳プロンプトは以下の方針で設計する。

1. **キャラクター名辞書**: 作品ごとにキャラクター名・固有名詞の英語表記を辞書として定義し、翻訳時に一貫性を保つ
2. **文体設定**: 「硬め（ハイファンタジー向け）」「柔らかめ（日常系向け）」「中間」の3パターンから選択可能
3. **一人称の使い分け**: 日本語の一人称（俺/僕/私/わたくし等）をキャラクターの性格に応じた英語表現に変換
4. **コンテキスト保持**: 前章の翻訳結果を参照し、名前の表記揺れや設定の矛盾を防ぐ
5. **ルビ対応**: ルビ（振り仮名）を英語の括弧注釈またはイタリック表記に変換
