## 1. 背景と目的

### 1.1 背景

**プロジェクトの背景:**

Discoveryフェーズにおいて、以下の6つのドキュメントを完成させ、Web小説のグローバル展開における課題を深く理解した。

1. **ユーザーリサーチ** — 作家・読者双方の生の声を収集し、翻訳コスト・品質・プラットフォームの課題を特定
2. **競合分析** — Syosetu/カクヨム/Webnovel/DeepL等の4社を分析し、「ラノベ特化AI翻訳×作家還元」の空白市場を発見
3. **ペルソナ** — 山田太郎（作家・プライマリ）、Emily Chen（海外読者・セカンダリ）、佐藤美咲（プロ作家・セカンダリ）の3名を定義
4. **共感マップ** — 各ペルソナの思考・感情・行動パターンを可視化し、痛みと期待を明確化
5. **ジャーニーマップ** — 「執筆→翻訳→公開→読者獲得→収益化」の全プロセスにおけるペインポイントをマッピング
6. **HMW（How Might We）** — 「どうすれば翻訳の壁を壊せるか」「どうすれば作家に正当な収益を届けられるか」「どうすれば読者の没入体験を実現できるか」の3つの問いを設定

**解決したい課題:**

| 課題 | 内容 | 影響 |
|------|------|------|
| 翻訳の壁 | 日本語Web小説の英語翻訳コストが1作品50〜100万円、期間6ヶ月以上 | 99%以上の作品が海外読者に届かない |
| 収益構造の歪み | 既存プラットフォームの印税率5〜15%、投げ銭手数料50%以上 | 作家の創作モチベーション低下 |
| プラットフォーム老朽化 | なろう/カクヨムのUI/UXが2010年代のまま、海外読者向け機能なし | グローバル展開の障壁 |

### 1.2 MVPの目的

**このMVPで検証したいこと:**
- [x] ラノベ特化AI翻訳が、ファンサブ翻訳と同等以上の品質（4.0/5.0以上）を実現できるか
- [x] なろう/カクヨムの作家が月¥1,000の翻訳サービスに支払意思を持つか
- [x] 海外ラノベ読者がAI翻訳作品を「十分に読める」と感じ、継続的に読むか

**期待される学び:**

- AI翻訳のどの側面（語彙の正確さ、文体の再現性、キャラクターの一人称、固有名詞の一貫性）が読者の没入感に最も影響するかを理解する
- 作家が翻訳サービスに求める最低限の機能セットを特定する
- 海外読者の作品発見から継続読書に至るユーザージャーニーを実データで検証する

## 2. 仮説定義

### 2.1 価値仮説（Value Hypothesis）

**誰が:** 山田太郎タイプ（なろう/カクヨムで連載中、累計10万PV以上、海外展開に興味があるがコスト障壁で諦めている作家）

**何に困っていて:** 翻訳コスト50〜100万円・期間6ヶ月以上の障壁により、自分の作品を海外読者に届けることができない。既存の機械翻訳（Google翻訳・DeepL）ではラノベ特有の文体（一人称の使い分け、異世界固有名詞、ルビ表現）が正しく翻訳されず実用に耐えない。

**何を提供すれば:** ラノベ特化AI翻訳エンジン（月¥1,000で無制限翻訳、ワンクリックで日→英翻訳、キャラクター名辞書・文体設定対応）

**どうなるか:** 作家が月¥1,000で自分の作品を即座に英語圏の読者に届けられるようになり、「自分の作品が世界で読まれている」という体験を得る。Emily Chenタイプの海外読者が「borderline unreadable」なファンサブ翻訳から「没入できる品質」の翻訳を得られるようになる。

**検証方法:**
- A/Bテスト: AI翻訳 vs 既存ファンサブ翻訳の品質比較（読者によるブラインド評価）
- 翻訳品質スコア: 5段階評価アンケート（目標4.0/5.0以上）
- 作家の初回翻訳完了率: 登録→翻訳実行までの完了率（目標40%以上）

### 2.2 成長仮説（Growth Hypothesis）

**どのチャネルで:** X（Twitter）の日本語Web小説作家コミュニティ、Reddit r/LightNovels・r/noveltranslations、Discord翻訳コミュニティ

**どのように顧客を獲得し:**
- 作家側: X（Twitter）でのβテスター募集、なろう/カクヨムの上位ランキング作家への直接DM、作家コミュニティでの口コミ
- 読者側: Reddit/Discordでの翻訳作品公開、英語圏ラノベレビューサイトとの連携

**どのように収益化するか:**
- 作家: 月額¥1,000のサブスクリプション（翻訳枚数無制限）
- 読者: 月額$5のサブスクリプション（全作品読み放題）
- 投げ銭: 30%プラットフォーム手数料、70%作家還元

**検証方法:**
- LP登録率: β版LP訪問→メール登録の転換率（目標5%以上）
- 作家アンケート: 月¥1,000の支払意思（目標60%以上が「払う」と回答）
- 読者無料→有料転換率: 無料期間後のサブスク加入率（目標5%以上）

### 2.3 最もリスクの高い仮説

**このMVPで最優先で検証すべき仮説:**

> ラノベ特化AI翻訳が、ファンサブ翻訳と同等以上の品質を実現し、海外読者の70%以上が「十分に読める」と評価する

**なぜこれが最もリスクが高いか:**

この仮説はastranoteのプロダクト価値の根幹を成す。AI翻訳の品質が不十分であれば、作家は翻訳結果を公開したがらず（供給サイドの崩壊）、読者は離脱する（需要サイドの崩壊）。価格や機能の問題はピボット可能だが、翻訳品質の問題はプロダクトの存在意義そのものに関わる。リーンキャンバスで特定した3つの仮説（AI翻訳品質、作家の支払意思、読者のサブスク加入）のうち、他の2つは価格調整やモデル変更で対応可能だが、翻訳品質の問題は技術的な根本課題であり、後からの修正コストが最も高い。

- 居住地: 日本国内（首都圏中心）

## 4. MVPタイプの選択

### 4.1 選択したMVPタイプ

**[ ] コンシェルジュ型** - 手動でサービスを提供
**[ ] ペーパーモック型** - 紙やスライドを使って反応を確かめる
**[ ] ランディングページ型** - LPで需要確認
**[ ] オズの魔法使い型** - 自動化に見せて手動対応
**[x] シングル機能型** - 1つの機能に絞った実装
**[ ] その他:**

### 4.2 選択理由

シングル機能型MVPを選択した理由は以下の3点である。

1. **翻訳品質は体験しなければ評価できない**: astranoteの最重要仮説は「AI翻訳品質がラノベ読者の没入感を満たせるか」である。これはランディングページやペーパーモックでは検証不可能であり、実際に動く翻訳エンジンを読者に体験させる必要がある。
2. **コンシェルジュ型では翻訳品質のスケーラビリティを検証できない**: 人間翻訳者による手動対応では、AI翻訳特有の品質課題（文脈理解、固有名詞の一貫性等）が見えない。
3. **プロンプトエンジニアリングの反復検証が必要**: ラノベ特化の翻訳品質を実現するには、プロンプトの調整→翻訳結果の評価→改善のBuild-Measure-Learnサイクルを高速で回す必要があり、動くプロダクトが不可欠。

### 4.3 期間と予算

**開発期間:** 10〜12週間（2026-02-17 〜 2026-04-27〜5-11）

> **スケジュール修正（技術レポート指摘）:** 当初計画の8週間は、翻訳エンジン（M2）の工数を過小評価していた。プロンプトエンジニアリングのチューニングサイクル（翻訳→評価→改善）には実測値で3〜4週間が必要と判断し、M2を2週間から3〜4週間に延長する。また、MVP開発前に2週間の翻訳品質POC（事前技術スパイク）を実施するため、全体スケジュールを10〜12週間に延長する。詳細は「[翻訳品質POC](#翻訳品質poc事前技術スパイク)」セクションを参照。

## 翻訳品質POC（事前技術スパイク）

> **技術レポート最優先提言:** MVP本開発に入る前に、2週間の集中的な翻訳品質POC（技術スパイク）を実施する。「AI翻訳品質がラノベ読者に受け入れられる」という最重要仮説を、本開発コストをかける前に検証する。POCで目標スコア（4.0/5.0）を達成した場合のみ、MVP本開発に移行する。

### POCの目的

本開発（10〜12週間）に入る前に、以下の技術的実現可能性を検証する。

1. **ラノベ特有の文体に特化したプロンプトでGPT-4/Claude APIが4.0/5.0以上を実現できるか**
2. **どのプロンプト戦略が最も高スコアを出すか**（比較評価）
3. **翻訳APIコストの実測値を取得する**（コスト構造の見直しに活用）

### POCの実施計画

**期間:** 2週間（MVP本開発前、2026-02-17 〜 2026-02-28）

**評価対象:**

| 評価軸 | 内容 |
|--------|------|
| 評価作品数 | 5作品（ジャンル分散: 異世界転生・ファンタジー・日常系・ラブコメ・スポーツ） |
| 評価話数 | 各作品5話（合計25話） |
| プロンプト戦略数 | 3〜5パターン（ベースライン・キャラクター辞書あり・文体指定あり・コンテキスト保持あり・組み合わせ）を比較 |
| 比較対象 | 同一作品の既存ファンサブ翻訳（ブラインドテスト用） |

**評価方法:**

| 評価方法 | 内容 | 人数 |
|---------|------|------|
| 英語読者によるブラインドテスト | AI翻訳とファンサブ翻訳を区別せずに提示し、5段階評価+コメントを収集 | 10名（最低限）推奨20名 |
| 翻訳専門家によるレビュー | ラノベ翻訳経験のある翻訳者2名による品質評価（語彙・文体・固有名詞・一貫性） | 2名 |
| 定量スコア集計 | 各プロンプト戦略の平均スコアを算出し、最優秀プロンプトを特定 | - |

> **サンプルサイズの注記（技術レポート P2 対応）:** n=10の評価では信頼区間が±14%程度と広く、スコア4.0は統計的に3.86〜4.14の範囲を意味する。Go/No-Go判定の信頼性を高めるために、可能であればn=20への拡大を推奨する。n=20の場合、信頼区間は±10%程度に縮小し、意思決定の確実性が向上する。テスター謝礼予算に余裕がある場合は優先的にn=20で実施する。

**成功基準（Go/No-Go判定）:**

| 判定 | 条件 | 次のアクション |
|------|------|--------------|
| Go（本開発へ移行） | 最優秀プロンプトの翻訳品質スコアが4.0/5.0以上 | 2026-03-01からMVP本開発を開始 |
| 要改善（条件付きGo） | スコア3.5〜3.9。追加プロンプト改善で4.0到達の見込みあり | 1週間の追加チューニング後に再評価 |
| No-Go（ピボット検討） | スコア3.5未満。プロンプト改善だけでは限界の可能性 | 人間翻訳者+AIのハイブリッドモデルへのピボットを検討 |

**POC期間中のコスト目安:**

| 項目 | 概算 | 備考 |
|------|------|------|
| AI翻訳API費用 | ¥10,000〜20,000 | 25話 × 5,000文字 × 5プロンプト × ¥0.5〜1.5/1,000文字 |
| テスター謝礼 | ¥15,000〜20,000 | 英語読者10名 × ¥1,500〜2,000 |
| 翻訳専門家費用 | ¥10,000〜30,000 | 翻訳者2名のレビュー報酬 |
| POC合計 | 約¥35,000〜70,000 | 本開発予算¥500,000の7〜14% |

### POC結果による計画への影響

POCの結果は、MVPの機能スコープ・技術スタック・予算配分に直接影響する。

| POC結果 | 影響 |
|---------|------|
| Go（スコア4.0以上） | 計画通りにMVP本開発を開始 |
| 要改善（3.5〜3.9） | M2（翻訳エンジン）の工数を+1週間追加、予算をAPIチューニングに+¥20,000配分 |
| No-Go（3.5未満） | ソリューションをピボット。コンシェルジュ型MVP（人間翻訳+AI補助）に切り替えを検討 |

## 5. 機能スコープ

### 5.1 MVPに含める機能（Must Have）

> **スコープ修正（批判・UXレポート指摘）:** MVPの最優先目標を「翻訳品質の検証」に絞り込む。投げ銭・高度なエディタ機能はβ版以降に移動。代わりに読者フィードバック収集UI（翻訳品質評価）とオンボーディングフロー設計をMVPスコープに追加する。

| 優先度 | 機能名 | 説明 | 検証する仮説 | 工数見積 |
|--------|--------|------|--------------|----------|
| P0 | ラノベ特化AI翻訳（日→英） | Claude API / GPT-4 APIを活用し、ラノベ特有の文体（一人称、固有名詞、ルビ）に対応した日英翻訳。キャラクター名辞書、文体設定（硬め/柔らかめ）をサポート。POCで最優秀と評価されたプロンプト戦略を採用 | 仮説1: AI翻訳品質 | 3〜4週間 |

### 5.2 MVPに含めない機能（Will Not Have）

**意図的に含めない機能:**
- 投げ銭/決済機能 — Stripe連携、作家への送金フロー
- マルチプラットフォーム同時投稿 — なろう/カクヨム/Webnovelへの自動投稿
- 読者コミュニティ機能 — コメント、フォロー、お気に入り、ランキング
- 多言語翻訳（英語以外） — 中国語、韓国語、スペイン語等への翻訳
- 高度なエディタ機能 — WYSIWYG、画像挿入、共同執筆
- 作家ダッシュボード — PV分析、読者属性分析、収益レポート
- SEO/ASO最適化 — 検索エンジン向けメタデータ最適化

**理由:**

MVPの目的は「AI翻訳品質の検証」に集中すること。決済・コミュニティ・分析等の機能は翻訳品質の仮説が検証された後に追加する。これらの機能をMVPに含めると、開発期間が3〜4倍に膨らみ、検証の焦点がぼやける。リーンキャンバスで定義した損益分岐点（ライター400名+読者2,000名）の達成は正式リリース後の目標であり、MVPフェーズでは追求しない。

### 5.3 次のイテレーションで検討する機能

- 投げ銭機能（Stripe連携、70%作家還元） — 仮説3（読者のサブスク加入）の検証後
- 読者ダッシュボード（PV、フォロワー数、読了率の可視化） — β版のデータを基に設計
- コミュニティ機能（コメント、フォロー） — 読者のエンゲージメント指標を見てから判断
- なろう/カクヨムからのワンクリックインポート — 作家のオンボーディング改善用
- 多言語翻訳（中国語、韓国語） — 英語翻訳の品質が確立された後に展開

### 6.1 定量指標

| 指標 | 現状 | 目標値 | 計測方法 | 備考 |
|------|------|--------|----------|------|
| 翻訳品質スコア | N/A（新規） | 4.0/5.0以上 | 読者アンケート（5段階評価） | 最重要指標 |

**AARRR指標:**
- **Acquisition（獲得）:**
  - 指標: β版LP訪問→メール登録の転換率
  - 目標: 5%以上（LP訪問者のうち）

- **Activation（活性化）:**
  - 指標: 登録→初回翻訳完了の転換率（作家）、登録→3話完読の転換率（読者）
  - 目標: 作家40%以上、読者30%以上

- **Retention（定着）:**
  - 指標: 7日後の再訪率（作家）、翌週の再読率（読者）
  - 目標: 作家25%以上、読者20%以上

- **Revenue（収益）:**
  - 指標: 「月¥1,000を払う意思がある」と回答した作家の割合（アンケート）
  - 目標: 60%以上（MVPフェーズでは実際の課金は行わない）
  - **重要な限界と注記（技術レポート P1-7 対応）:** β版ではStripeが未実装のため、支払意思の検証はアンケートベースに限られる。「バベル効果」（口頭では払うと言うが実際には払わないバイアス）のリスクがあり、アンケート結果の60%達成は、実際の課金転換率を保証しない。正式リリース後の実課金データによる再検証が必須。アンケートWTPは参考値として扱い、意思決定の主要根拠にはしない。

- **Referral（紹介）:**
  - 指標: 翻訳作品のURLシェア数、X（Twitter）でのastranote言及数
  - 目標: β版参加者の20%以上がSNSでシェア

### 6.2 定性指標

**観察したいユーザー行動:**
- 作家が翻訳結果を見た瞬間の反応（「おお！」「すごい！」等の感嘆の声があるか）
- 作家が翻訳結果をそのまま公開するか、それとも手動で修正してから公開するか（修正箇所の特定）
- 読者がAI翻訳作品を読み始めてから離脱するまでの読書パターン（どの章で離脱するか）
- 読者がファンサブ翻訳とAI翻訳を比較した時の具体的なフィードバック
- 作家同士でastranoteについて会話しているか（自然発生的な口コミ）

### 6.3 成功/失敗の判断基準

**成功と判断する条件:**
- 翻訳品質スコアが4.0/5.0以上（読者の70%以上が「十分に読める」と評価）
- 初回翻訳完了率が40%以上（作家のアクティベーション）
- 3話完読率が30%以上（読者のエンゲージメント）
- 作家の60%以上が「月¥1,000を払う」と回答

**失敗と判断する条件:**
- 翻訳品質スコアが3.0/5.0未満（読者の過半数が「読めない」と評価）
- 初回翻訳完了率が20%未満（作家がプロダクトを使いこなせない）
- 3話完読率が10%未満（読者が離脱する）
- 作家の支払意思が30%未満

**判断のタイミング:** β版開始4週間後（2026-06-09を予定）

## 7. 検証計画（MVPキャンバス）

### 7.1 仮説

**最も重要な仮説:**

> ラノベ特化AI翻訳（プロンプトエンジニアリングによるキャラクター名辞書・文体設定・コンテキスト保持）が、既存のファンサブ翻訳と同等以上の品質を実現し、海外ラノベ読者の70%以上が「十分に読める」と評価する。

### 7.2 リスク

**想定されるリスク:**

| リスク | 影響度 | 発生確率 | 対応策 |
|--------|--------|----------|--------|
| 1. AI翻訳品質がラノベ読者の没入感を満たせない | 最高 | 中 | 人間翻訳者とのハイブリッドモデルへピボット |
| 2. 作品ラインナップ不足（テスター作家の作品が少ない） | 高 | 中 | テスター数を50名に増員、既存人気作品の許諾取得 |
| 3. 技術的スケーラビリティ（API呼び出し制限、レスポンス遅延） | 中 | 低 | APIプロバイダの分散（Claude + GPT-4）、キャッシュ戦略 |
| 4. テスター離脱（β版期間中の離脱率が高い） | 中 | 中 | 週次フォローアップ、謝礼の段階支給 |
| 5. 競合の先行リリース（Webnovel等が類似機能をリリース） | 低 | 低 | ラノベ特化の深さで差別化、速度よりも品質を優先 |

**対応策:**
- リスク1（翻訳品質不足）: Week 3-4でプロンプトの集中的なチューニングを実施。品質が目標に達しない場合、人間翻訳者によるポストエディット（AI翻訳→人間校正）のハイブリッドモデルを検討
- リスク2（ラインナップ不足）: 事前に5作品以上のテスト用コンテンツを確保。既存のCC-BY作品やテスター自身の作品を活用
- リスク3（スケーラビリティ）: 翻訳結果のキャッシュ、バッチ処理の導入。APIの並列呼び出しでレスポンス改善

## 8. 開発計画

### 8.1 技術スタック

**フロントエンド:** Next.js 15 (App Router) + TailwindCSS + shadcn/ui

**バックエンド:** Supabase (Auth / PostgreSQL / Storage / Edge Functions)

**AI翻訳:** Claude API (Anthropic) / GPT-4 API (OpenAI)（ラノベコーパスでのプロンプトエンジニアリング）

**選定理由:**

個人〜少人数チームでのMVP開発に最適化した技術スタックを選定した。

| 開発速度 | Next.js + Supabase の組み合わせにより、認証・DB・ストレージをほぼゼロコンフィグで利用可能。フルスクラッチ比で開発期間を50%短縮 |

| AI翻訳品質 | Claude API / GPT-4 API は日本語→英語の翻訳品質が高い。プロンプトエンジニアリングにより、ラノベ特化のカスタマイズが可能 |

### 8.4 技術的な制約・前提

**制約:**
- AI翻訳APIのレート制限（Claude API: 1分あたり60リクエスト、GPT-4 API: 1分あたり500リクエスト）
- 1章あたりの翻訳レスポンスタイム目標10秒以内（5,000文字程度を想定）
- Supabase無料プランのDB容量500MB（MVP期間中は十分）
- 個人開発のため、24時間対応のサポート体制は構築しない

**前提:**
- AI翻訳APIの料金体系がMVP期間中に大幅に変更されないこと
- なろう/カクヨムの利用規約が、外部サービスへのテキストコピーを禁止していないこと
- β版テスターが十分な数（作家30名・読者100名）集まること
- ラノベ特化のプロンプトエンジニアリングにより、汎用翻訳を超える品質が実現可能であること

### 8.5 技術的リスクと対策

MVP開発における技術的リスクと、各リスクへの対策を以下に定義する。これらのリスクは、翻訳品質POCの結果や開発進行状況に応じて継続的に更新する。

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|----------|------|
| **リスク1: AI翻訳品質の不確実性**（ラノベ特有文体への対応不足） | 最高 | 中 | **POCで事前検証**（MVP本開発前に2週間のスパイクを実施し、4.0/5.0以上を確認してから着手。POCがNo-Goの場合は人間翻訳者+AIのハイブリッドモデルへピボット） |
| **リスク2: APIコスト変動**（翻訳コストが想定を超える） | 高 | 中 | **文字数上限の設定**（MVP期間は月間翻訳文字数に上限を設定。超過分は作家に通知）または**パススルー設計**（翻訳コストを作家側に実費請求するモデルへの変更を検討） |
| **リスク3: なろう/カクヨム利用規約リスク** | 高 | 低〜中 | **法務レビューの実施**（MVP開発開始前に専門家によるレビュー実施）＋**作家自身のコピペフロー設計**（自動スクレイピングは行わず、作家が自身でテキストをペーストするフローのみ提供。スクレイピング機能は実装しない） |
| **リスク4: 縦書きCSS対応の複雑さ** | 中 | 高 | **MVP期間は横書きのみ**に限定。縦書きは日本の読者体験には重要だが、英語読者向けには横書きが標準であるため、MVP段階での影響は限定的。縦書き対応はβ版以降に計画する |
| **リスク5: レスポンスタイムが目標を超える** | 中 | 中 | **バッチ翻訳＋キャッシュ戦略**を導入。翻訳リクエスト送信後にバックグラウンドで処理し、完了時に通知する非同期フローを採用。リアルタイム翻訳（10秒以内）が困難な場合は、「翻訳中」の待機UXを設計する |
| **リスク6: β版テスターの離脱** | 中 | 中 | 週次のフォローアップ（プログレスメール+Discord通知）と、段階的なテスター謝礼支給（翻訳完了数に応じた報酬）で継続率を維持 |

### 8.6 法的前提条件

> **MVP開発開始前に確認すべき法的事項。** このチェックリストを全て確認してから開発を開始する。未確認のまま開発を進めると、リリース後に法的問題が発覚して作業が無駄になるリスクがある。

**MVP開発開始前（2026年3月初旬まで）に確認すること:**

- [ ] **なろう/カクヨム利用規約の確認**: 作家が自身のテキストをコピーして外部サービスに入力することが利用規約に違反しないかを法務専門家が確認する
- [ ] **astranoteの利用規約・プライバシーポリシーの作成**: MVP α版リリース時点で有効な利用規約を作成。「アップロードできるのは作家自身が権利を保有するコンテンツのみ」の条項を明記する
- [ ] **AI翻訳生成物の著作権帰属の確認**: 翻訳物の著作権が作家に帰属することを利用規約で明確化する。各国（日本・米国）の法律上の問題がないか確認する
- [ ] **個人情報保護法への対応**: β版テスターの個人情報（メールアドレス等）の取扱い方針を策定する（日本の個人情報保護法・EU GDPRへの対応を確認）

**β版リリース前（2026年4月までに）確認すること:**

- [ ] **書籍化作家の契約確認フロー**: 書籍化経験のある作家がβ版に参加する場合、出版社との契約において海外出版権に制限がないかをオンボーディング時に確認するフローを設ける
- [ ] **Stripe決済の利用規約確認**: 将来の投げ銭機能実装に向けて、Stripeの利用可能な事業形態と制限事項を事前確認する（β版では実装しないが、設計段階で考慮）

### 8.7 品質基準

**最低限の品質基準（横書き読書ビューア前提）:**
- [x] 翻訳レスポンスタイム: 5,000文字あたり10秒以内（非同期フロー採用時は「翻訳完了通知」方式でも可）
- [x] サービス稼働率: 99.5%以上（計画メンテナンス除く）
- [x] アクセシビリティ: WCAG 2.1 AA準拠（読書ビューアのフォントサイズ・コントラスト）
- [x] モバイルレスポンシブ: iOS Safari、Android Chrome での読書体験を保証
- [x] データ保護: 作家の原稿データの暗号化保存、HTTPS通信の強制

**テスト方針:**
- ユニットテスト: 翻訳エンジンのコア機能（文字数カウント、章分割、辞書適用）
- E2Eテスト: 主要ユーザーフロー（登録→執筆→翻訳→公開→読了）のPlaywrightテスト
- 翻訳品質テスト: 10作品×3章=30パターンの翻訳結果を人間評価で品質チェック
- 負荷テスト: 同時翻訳リクエスト30件でのレスポンスタイム検証

## 9. リリース計画

### 9.1 リリーススケジュール

**POC（翻訳品質スパイク）:** 2026-02-17 〜 2026-02-28（2週間）
- 対象: 5作品×5話のサンプルテキスト、英語読者10名（クラウドソーシング等で募集）
- 目的: AI翻訳品質のGo/No-Go判定。4.0/5.0以上を確認してから本開発へ移行
- 期間: 2週間

**α版（社内テスト）:** 2026-04-27（Week 9〜10 相当）
- 対象: 社内メンバー + 親しい作家 5名
- 目的: 致命的なバグの発見、翻訳品質の実装確認、UXの主要課題の特定（特にオンボーディングフロー）
- 期間: 2週間（2026-04-27 〜 2026-05-11）

**β版（限定公開）:** 2026-05-12（Week 11〜13 相当）
- 対象: アーリーアダプター作家 30名 + 読者 100名
- 目的: 仮説検証（翻訳品質スコア、初回翻訳完了率、3話完読率、支払意思）
- 期間: 4週間（2026-05-12 〜 2026-06-09）

**正式リリース:** 2026-07-07（β版検証成功判断後4週間）
- 対象: 一般公開（作家・読者ともに制限なし）
- 目的: 損益分岐点に向けたスケール開始（ライター400名+読者2,000名を12ヶ月で達成）
- 前提: β版検証で成功基準を満たした場合のみ

> **スケジュール修正の背景:** POC（2週間）、M2翻訳エンジンの工数延長（3〜4週間）、M3 UI/UX実装の工数延長（3〜4週間: 執筆エディタ2W + 公開機能1.5W + 評価UI1W + オンボーディング1W）、新規P0機能（翻訳品質評価UI・オンボーディングフロー）の追加により、全体スケジュールが当初の8週間からPOC+M1〜M4合計で12〜15週間に延長される。

### 9.3 ロールバック計画

**緊急時の対応:**

Vercelのインスタントロールバック機能を活用し、前回の安定版に即座に切り戻す。翻訳APIの障害時は、多層フォールバック設計（Claude API → GPT-4 API → DeepL API → Google Translate API）で自動切り替えし、Claude/GPT-4同時障害時にも翻訳サービスを継続できる仕組みを実装する。

> **フォールバック設計の補足（批判レポート 問題3 対応）:** Claude/GPT-4の同時障害リスクに備え、DeepL APIとGoogle Translate APIを第2・第3フォールバックとして追加する。ただし、DeepL/Google Translateはラノベ特化プロンプトが適用されないため、フォールバック時の翻訳品質は主系（Claude/GPT-4）より低下する可能性があることをユーザーへ通知する。

**ロールバック条件:**
- サービス稼働率が95%を下回った場合（24時間以内）
- 翻訳APIの連続エラー率が10%を超えた場合
- ユーザーデータの損失・漏洩が確認された場合（即時ロールバック+サービス停止）

## 12. 意思決定

> **注記:** このセクションはβ版検証期間終了後（2026-05-11以降）に、検証結果に基づいて記録する。以下に事前定義した判断基準を記載しておく。

### 12.1 次のアクション

**[ ] 継続（Persevere）** - 現在の方向性を続ける
**[ ] ピボット（Pivot）** - 方向性を転換する
**[ ] 中止（Stop）** - このアイデアは諦める

### 12.2 判断理由

**事前定義した判断基準:**

| 判断 | 条件 |
|------|------|
| **継続** | 翻訳品質スコア4.0以上 AND 初回翻訳完了率40%以上 AND 作家支払意思60%以上 |
| **ピボット** | 翻訳品質スコア3.0〜3.9 OR 初回翻訳完了率20〜39% OR 作家支払意思30〜59% |
| **中止** | 翻訳品質スコア3.0未満 AND 作家支払意思30%未満 AND 改善の見込みなし |

### 12.3 継続する場合

**次のイテレーションの焦点:**
- 投げ銭/決済機能の実装（Stripe連携、70%作家還元）
- 読者コミュニティ機能（コメント、フォロー、お気に入り）
- 多言語翻訳の追加（中国語・韓国語）

**追加する機能:**
- Stripe決済（作家月額¥1,000、読者月額$5）
- 読者ダッシュボード（お気に入り管理、読書履歴）
- 作家ダッシュボード（PV分析、読者属性、収益レポート）

**改善する点:**
- 翻訳品質のさらなる向上（フィードバックに基づくプロンプト改善）
- UI/UXの改善（β版テスターのフィードバック反映）
- パフォーマンス最適化（翻訳レスポンスの高速化）

### B. 翻訳品質評価基準

| スコア | 評価 | 基準 |
|--------|------|------|
| 5 | 優秀 | プロの翻訳者と遜色ない品質。キャラクターの口調、世界観の雰囲気が忠実に再現されている |
| 4 | 良好 | 多少の不自然さはあるが、没入感を損なわずに読み通せる。ファンサブ翻訳以上の品質 |
| 3 | 普通 | 意味は通じるが、文体の不自然さが目立つ。読了はできるが没入感に欠ける |
| 2 | 不足 | 誤訳や不自然な表現が多く、読み進めるのに苦労する。ファンサブ翻訳以下の品質 |
| 1 | 不可 | 意味が通じない箇所が多く、読了が困難。実用に耐えない |

### C. プロンプトエンジニアリング方針

MVP期間中のAI翻訳プロンプトは以下の方針で設計する。

1. **キャラクター名辞書**: 作品ごとにキャラクター名・固有名詞の英語表記を辞書として定義し、翻訳時に一貫性を保つ
2. **文体設定**: 「硬め（ハイファンタジー向け）」「柔らかめ（日常系向け）」「中間」の3パターンから選択可能
3. **一人称の使い分け**: 日本語の一人称（俺/僕/私/わたくし等）をキャラクターの性格に応じた英語表現に変換
4. **コンテキスト保持**: 前章の翻訳結果を参照し、名前の表記揺れや設定の矛盾を防ぐ
5. **ルビ対応**: ルビ（振り仮名）を英語の括弧注釈またはイタリック表記に変換
