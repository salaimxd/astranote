## Docs-as-Code × AI の基本思想

### コードのようにドキュメントを扱う

**従来のドキュメント作成:**
```
書く → 完成 → 放置 → 陳腐化
```

**Docs-as-Code × AI:**
```
生成 → レビュー → 反復改善 → バージョン管理 → 自動検証 → 継続的更新
```

### 5つの原則

1. **Single Source of Truth**: ドキュメントは唯一の真実の情報源
2. **Version Control**: すべての変更をGitで管理
3. **Automated Validation**: AIで品質を自動検証
4. **Iterative Refinement**: 反復的に改善
5. **AI as Pair Writer**: AIをペアライターとして活用

## 開発フェーズ別ワークフロー

### Phase 0: プロジェクト立ち上げ（初回のみ）

#### ゴール: プロダクトの基盤ドキュメントを確立する

```
Vision/Mission → Persona → Roadmap → Messaging House
```

#### ワークフロー

**Step 1: AIにコンテキストを提供**

```bash
# Claude/ChatGPTに以下を提供
1. [00.04_AI_Context_Index.md](00.04_AI_Context_Index.md) をコピー
2. プロダクト情報を簡潔に説明（2-3文）
   - 何を作るか
   - 誰のためか
   - なぜ必要か
```

**プロンプト例:**
```
以下のプロジェクトのVision/Missionドキュメントを作成してください：

【プロダクト概要】
- プロダクト名: WorkflowHub
- ドメイン: B2B SaaS
- 提供価値: チームのワークフローを自動化し、生産性を30%向上させる

【ターゲット】
- スタートアップ〜中小企業のプロジェクトマネージャー
- 現状の課題: 複数ツールの手動連携が煩雑

テンプレート: /home/salaim/work/product-docs/10_Strategy_Planning/11.01_Vision_Mission.md
コンテキスト: /home/salaim/work/product-docs/00_Meta/00.04_AI_Context_Index.md
```

**Step 2: AIの出力をレビュー・編集**

```bash
# 生成されたドキュメントを保存
vim 10_Strategy_Planning/11.01_Vision_Mission.md

# レビューポイント
✓ プロダクトの本質が表現されているか
✓ チーム全員が共感できるか
✓ 測定可能なゴールが含まれているか
✓ 顧客価値が明確か
```

**Step 3: Gitでバージョン管理開始**

```bash
git add 10_Strategy_Planning/11.01_Vision_Mission.md
git commit -m "Vision/Mission: プロダクトの方向性を定義 v1.0"
git push origin main
```

**Step 4: 他の基盤ドキュメントを連鎖的に作成**

```bash

- Vision/Mission: [11.01_Vision_Mission.md]
- Persona: [21.02_Persona_Project_Manager.md]

テンプレート: 10_Strategy_Planning/12.01_Roadmap.md」
```

### Phase 1: Discovery（調査・仮説検証）

#### ゴール: ユーザーニーズを理解し、解決すべき課題を特定する

```
User Interviews → Persona → Market Trends → Churn Analysis
```

#### AI駆動ワークフロー

**シナリオ: 5件のユーザーインタビューを実施した**

**Step 1: インタビュー結果をAIで要約**

```bash
# インタビュー音声を文字起こし（Whisper、Otter.ai等）
# 生データ: interviews_raw/interview_01.txt 〜 05.txt

# AIプロンプト
「以下の5件のユーザーインタビュー結果を分析し、
User Interviews Summaryを作成してください：

【分析観点】
- 共通するペインポイント
- 頻出する要望（3回以上言及）
- 現在の代替手段（競合、手動オペレーション）
- ユーザーの優先順位

【インタビュー内容】
[interview_01.txt 〜 05.txt を貼り付け]

テンプレート: 20_Discovery_Research/21.01_User_Interviews_Summary.md
プロンプト参照: 00_Meta/00.03_AI_Prompts.md#persona作成
」
```

**Step 2: AIが生成した要約を人間がレビュー**

```bash
# チェックポイント
✓ 生のインタビューデータと矛盾していないか
✓ 重要な洞察が欠けていないか
✓ バイアスがかかっていないか（例: AIが勝手に解釈を加えていないか）
✓ 定量的なデータ（頻度、割合）が含まれているか
```

**Step 3: Personaを反復的に改善**

```bash
# 既存Personaを読み込んでAIに修正依頼
「以下の新しいUser Interviews結果を踏まえて、
既存のPersonaを更新してください：

【既存Persona】
[21.02_Persona_Project_Manager.md]

【新しい洞察】
[21.01_User_Interviews_Summary.md の追加内容]

【更新指示】
- ペインポイントに「複数ツール間のコンテキストスイッチ」を追加
- 意思決定プロセスに「セキュリティ承認が必須」を追加
」
```

**Step 4: 差分をGitで管理**

```bash
# 変更を確認
git diff 20_Discovery_Research/21.02_Persona_Project_Manager.md

# コミット
git add 20_Discovery_Research/21.02_Persona_Project_Manager.md
git commit -m "Persona: インタビュー結果（5件）を反映してペインポイントを更新"
git push
```

## AIとのペアプログラミング的なドキュメント作成

### 「ドライバー」と「ナビゲーター」の役割分担

**人間 = ドライバー（意思決定者）**
- プロダクトの方向性を決める
- 重要な意思決定を行う
- ドキュメントの最終レビュー
- AIの出力を評価・編集

**AI = ナビゲーター（提案者・検証者）**
- ドラフトを生成
- 品質チェック
- 整合性検証
- 代替案の提示

### 反復サイクル

```
1. 人間: AIに指示（コンテキスト + プロンプト）
2. AI: ドラフト生成
3. 人間: レビュー・修正指示
4. AI: 修正版生成
5. 人間: 承認 or 再修正
   → 承認なら次のセクションへ
   → 再修正なら Step 3 へ戻る
```

### 効率的なプロンプト作成テクニック

#### テクニック1: コンテキスト階層化

```
【レイヤー1: プロジェクト全体】
00.04_AI_Context_Index.md（常に提供）

【レイヤー2: 関連ドキュメント】
Vision/Mission、Persona、Roadmap（該当箇所のみ）

【レイヤー3: 具体的な指示】
今回作成するドキュメントの詳細
```

#### テクニック2: 制約と期待値の明示

```
「以下の制約を守ってください：
- 文字数: 2000文字以内
- トーン: フォーマル（社内向け）
- 対象読者: エンジニア
- 必須項目: セキュリティ考慮事項は必ず3つ以上

期待する出力:
- Markdown形式
- テンプレートの全セクションを埋める
- プレースホルダーを残さない
」
```

#### テクニック3: Few-Shot Learning（例示）

```
「以下の例のような形式で出力してください：

【良い例】
## 3. 目標
### ビジネスゴール
- **主目標**: SSO利用率を6ヶ月で50%に到達
- **副次的な目標**: サポート問い合わせ（パスワード関連）を80%削減

### 成功指標 (KPI)
| 指標 | ベースライン | 目標 | 測定方法 |
| --- | --- | --- | --- |
| SSO利用率 | 0% | 50% | Analytics Dashboard |

【悪い例】
## 3. 目標
SSO機能を多くのユーザーに使ってもらう。
」
```

## 品質保証とレビュー自動化

### 自動化レベル

| レベル | 内容 | ツール | 所要時間 |
| --- | --- | --- | --- |
| Level 0 | 人間のみ | - | 2-4時間 |
| Level 1 | AI品質レビュー | Claude/ChatGPT | 30分 |
| Level 2 | AI整合性チェック | Claude/ChatGPT | 15分 |
| Level 3 | Markdown Linting | markdownlint-cli | 1分 |
| Level 4 | リンク切れチェック | markdown-link-check | 2分 |
| Level 5 | CI/CD統合 | GitHub Actions | 自動 |

### Level 1-2: AIレビューの実践

**1. 品質レビュー（各ドキュメントのAIレビューチェックリストを使用）**

```bash
# PRDの例
cd 30_Definition_Requirements
cat 31.01_PRD_SSO.md | pbcopy

# Claude/ChatGPTに貼り付け + プロンプト
「以下のPRDを品質レビューしてください：

[貼り付け]

チェック項目:
- 明確性: 目的とスコープが明確か
- 測定可能性: KPIが定量的か
- 完全性: エッジケースが考慮されているか
- 整合性: 他ドキュメントとの矛盾がないか
- セキュリティ: セキュリティリスクが洗い出されているか
- 実装可能性: エンジニアが実装を開始できるレベルか
」
```

**2. 整合性チェック（ドキュメント間の矛盾を検出）**

```bash
# PRD vs Tech Spec
「以下のPRDとTech Specの整合性をチェックしてください：

【PRD】
[31.01_PRD_SSO.md]

【Tech Spec】
[51.01_TechSpec_SSO.md]

矛盾があれば具体的に指摘してください。
プロンプト: 00_Meta/00.03_AI_Prompts.md#整合性チェック
」
```

### Level 3-4: ローカルでの自動化（オプション）

```bash
# Markdown Linting（ローカル実行）
npm install -g markdownlint-cli
markdownlint '**/*.md' --fix

# リンク切れチェック
npm install -g markdown-link-check
find . -name '*.md' -exec markdown-link-check {} \;
```

### Level 5: GitHub Actionsでの自動化（オプション）

```yaml

name: Docs Quality Check

on: [pull_request]

jobs:

```

## トラブルシューティング

### 問題1: AIの出力が期待と異なる

**原因:**
- コンテキスト不足
- プロンプトが曖昧

**解決策:**
```
1. 00.04_AI_Context_Index.md を必ず提供
2. 関連ドキュメントを全て貼り付け
3. 具体的な例を示す（Few-Shot Learning）
4. 制約と期待値を明示
```

### 問題2: AIが幻覚（Hallucination）を起こす

**原因:**
- AIが存在しないデータを創作

**解決策:**
```
1. 「既存の情報のみを使用してください」と明示
2. 数値データは必ず出典を確認
3. 人間が最終レビューで検証
4. AIの出力を「ドラフト」として扱う
```

### 問題3: ドキュメント間の整合性が取れなくなった

**原因:**
- 複数人が並行編集
- 一部のドキュメントのみ更新

**解決策:**
```
1. 変更前にAIで影響分析
   「このPRDを変更した場合、影響を受けるドキュメントは？」

2. 変更後にAIで整合性チェック
   「PRDとTech Specの整合性をチェック」

3. Gitでブランチを切って変更を隔離
   git checkout -b feature/prd-sso-update
```

## 次のステップ

1. **まずは1つのドキュメントをAIと一緒に作成してみる**
   - 簡単なものから（例: Release Notes）
   - プロンプトテンプレート（00.03_AI_Prompts.md）を活用

2. **反復サイクルを体験する**
   - AIドラフト → 人間レビュー → AI修正 → 承認

3. **チームで振り返り**
   - どのプロンプトが効果的だったか
   - どのセクションが人間が書くべきか
   - AIの精度はどうだったか

4. **徐々に自動化レベルを上げる**
   - Level 1-2（AIレビュー）から開始
   - 慣れたらLevel 3-4（Linting、Link Check）
   - 余裕があればLevel 5（CI/CD統合）
